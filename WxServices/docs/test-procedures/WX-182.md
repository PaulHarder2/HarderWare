# WX-182 — post-deploy verification (degrade circuit-breaker)

WX-182 closes the amplifier half of the WX-180 cost incident. A locality that
**degrades-to-no-send** on a given input (Claude can't produce a self-consistent
narrative and there's no near-term severe block) records that input on
`LocalityState.LastDegradedInputHash` ("WX-182 breaker armed …"). When a due
scheduled/first slot recurs on that **same** input, the breaker:

- **re-issues** the locality's last good report from cache — cached snapshot + its
  narrative with the "What's changed" band stripped + **live** current conditions —
  rendered as a `Scheduled` report with **no Claude call** ("re-issuing the … report
  from the last good snapshot"); or
- **skips** when there's no usable cached snapshot (e.g. a first-ever-degrade
  locality), self-healing when the input next changes.

The cached send leaves the breaker armed, so successive scheduled slots on the same
stuck input all serve from cache at **zero Claude cost**; a genuine Claude-reconciled
send (normal or degraded-hazard) clears it.

## This is a safety net — verification is preconditioned

WX-180 prong A already removed the main non-convergence cause, so the breaker may
legitimately **not fire at all** in a given window. The automated verdict is therefore
**preconditioned on a degrade-to-no-send having occurred** (the breaker armed). With no
degrade in the window there is nothing to verify → **WAIT**, not a false PASS.

Version-pinned to **1.30.0 / WxReportSvc** via `deploy-info.sh`; reads
`wxreport-svc.log`; shares the `verify-lib.sh` scaffold.

- **Precondition:** ≥1 `WX-182 breaker armed` line (a degrade-to-no-send). 0 → WAIT.
- **Fingerprint (expect, once armed):** `WX-182 degrade circuit-breaker …` lines
  (re-issue-from-cache and/or skip).
- **Failure signature → FAIL:** any `failed to render or send cached scheduled report`
  line — the cached send-from-snapshot path threw.
- **Health:** no new ERRORs vs the equal-length pre-deploy window.

## Manual confirmation (the definitive proof)

The breaker's purpose is cost control, so the conclusive evidence is on the **Grafana
WxReport cost panel**: during a non-convergence episode (a locality stuck degrading),
the daily Claude `$`-rate should stay **flat** rather than spiking as it did in the
WX-180 incident — because the stuck locality's per-cycle cost is now ~0 (pre-filter
skips between slots; cached sends at slots). Cross-check the token log: the stuck
locality should show `WX-182 degrade circuit-breaker` lines, **not** repeated
`generating scheduled report` → degrade every cycle.

If no natural episode occurs, the breaker can be exercised deliberately by inducing a
reproducible degrade (e.g. the WX-165 time-word case) on a test locality and confirming
the second identical-input scheduled slot re-issues from cache with no Claude call.
