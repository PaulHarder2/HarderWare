# WX-203 — es localized time at sentence end rendered a double period

Shipped in **1.34.2**. Verified by `WX-203-verify.sh` (DB scan) on top of deterministic unit tests.

## The bug

The es localized time ends in a period (`9:00 p. m.`). When Claude's es prose ends a sentence
right after a `{q:time}` token, `ReportTokens.Substitute` produced a **doubled period** —
`…despejando tras 9:00 p. m..` — the sentence period colliding with the abbreviation's own period.
Surfaced in WX-171's CodeRabbit review; the golden corpus was reworded to avoid baking it, but the
production behavior remained for real es recipients (paul_es / alex_es). en is unaffected
(`9:00 PM` has no trailing period).

## The fix

`ReportTokens.Substitute` (`StructuredReportBody.cs`) — the single chokepoint every prose render
flows through (one caller: the renderer's `RenderProse`) — now collapses a doubled period after
token expansion:

```csharp
var expanded = BraceSpanPattern().Replace(text, m => { …token expansion… });
return DoubledPeriodPattern().Replace(expanded, ".");   // (?<!\.)\.\.(?!\.)
```

The `(?<!\.)\.\.(?!\.)` source-generated regex matches only an **isolated pair**, so an intentional
ellipsis (`...`) is left intact. Spanish typography treats the abbreviation's trailing period as
also terminating the sentence, so collapsing to one period is correct. Language-general (en never
triggers it); at the chokepoint, so both the closing and the change band get it.

Unit tests (`StructuredReportBodyTests.Substitute_*`): es time at sentence end collapses, ellipsis
preserved, en unaffected, period-token mid-sentence untouched.

## Post-deploy verification (live)

`WX-203-verify.sh` (pinned to **1.34.2**), since the deploy boundary:

- **Failure signature** — *no* delivered body carries the doubled-period artifact `m..` (the es
  designator tail `p. m.` / `a. m.` doubled; ASCII-safe, robust to the NNBSP inside the designator).
  Post-fix this should be **0**; any occurrence ⇒ **FAIL**.
- **Gate** — no time window (deterministic fix); PASS needs **≥ 1 delivered es report** since the
  deploy (the fix is es-relevant) and zero artifacts. This is the *same* es-report precondition
  WX-171's verify waits on, so it adds **no new wait**.
- **The unit tests are the real proof.** The triggering collision (an es sentence ending right on a
  time token) is rare, so a clean es body can pass without exercising the exact case — the live scan
  guards against a binary/regex regression, not the transform itself.

**Run** (bash/WSL, after deploy): `docs/test-procedures/WX-203-verify.sh`. Same standing delegation
as the other verify scripts — run once ≥ 1 es report has accrued, record the Test Result, move
WX-203 to Done on PASS.
