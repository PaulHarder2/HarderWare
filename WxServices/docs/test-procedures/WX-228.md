# WX-228 — deterministic temperature-range summary (no more "the upper 97°F range")

**Type:** Functional verification (post-deploy). Scripted DB scan of persisted narratives (`WX-228-verify.sh`) plus a visual read of a delivered report.

**Release:** 1.38.2
**Components:** `WxReportSvc` — `TemperatureRangeSummarizer` (new), `ForecastReconciler.BuildUserMessage` (injects the computed ranges), `ReconcilerPrompts` (the `temp_range` guidance), `StructuredReportBody`/`ReportTokens` (the `{q:temp_range:lo:hi}` grammar), `StructuredReportRenderer.FormatTempRangeC` (the converted span).

## Background

The reconciler had only a single-**point** temperature token (`{q:temp:NN}`, canonical °C). Summarizing a week of highs/lows, the model wrapped a point in a fuzzy band word — "well into the upper `{q:temp:36}` range" — which rendered as the meaningless "the upper 97°F range". The fix adds a unit-neutral **range** token `{q:temp_range:lo:hi}` (one converted span — "97–99°F" / "36–37°C") and computes the one or two ranges deterministically (`TemperatureRangeSummarizer`: split into an early/later pair only when the halves' midpoints separate by ≥ 3 °C; otherwise one range — each the exact rounded °C min–max, a dead-flat run widened ±1 °C). The per-day high/low is the same max/min-across-blocks the Extended Forecast grid uses, so the prose and the grid agree.

## Preconditions

- 1.38.2 deployed (`deploy-history.log` carries a `WxReportSvc 1.38.2` line).
- At least one delivered report since the deploy whose closing summarizes temperatures — i.e. a scheduled/first send for a locality with a multi-day forecast. A non-English recipient (`*_es` / `*_de`) is desirable for the natural-prose spot-check.

## Procedure

1. **Run the scripted scan.**

   ```bash
   bash docs/test-procedures/WX-228-verify.sh
   ```
   - **Expect (PASS):** ≥ 1 delivered report since the deploy used a `{q:temp_range}` token (`sends using a {q:temp_range} token >= 1`) and **zero** band-wrapped point-token signatures. The script prints the most recent adopting report's per-language closings.
   - **WAIT** if no report has summarized temperatures yet (precondition absent, or the path not yet exercised) — re-run after the next active cycle.
   - **FAIL** if any narrative still wraps a `{q:temp:}` point in a band word (`upper {q:temp:…}`, `{q:temp:…} range`, `{q:temp:…}s`) — the old binary is live or the model reverted; check the deploy and the `ReconcilerPrompts` guidance.

2. **Read a delivered report (the human check the scan can't make).** Open a recent report (or the closings the script dumped).
   - **en — Expect:** highs/lows read as a clean span, e.g. "highs near 97–99°F" or "in the upper 90s"-class natural phrasing, and **never** "the upper NN°F range" or "in the low NNs" wrapped around a single value.
   - **A warming/cooling week — Expect:** at most two periods, e.g. "in the mid-20s°C early in the week, rising to the upper 20s later" — the split present only when the change is ≥ ~5 °F.
   - **es / de — Expect:** the same ranges, converted to each recipient's unit, phrased **natively and idiomatically** (e.g. de "36–37 °C", es "de 36 a 37 °C" / "en torno a…") — not a calque of the English guidance ("earlier part of the period").

3. **Cross-check prose vs grid.** Confirm the high/low range stated in the closing matches the day-by-day highs/lows in the Extended Forecast grid of the **same** report (they derive from the same per-day aggregation).
   - **Expect:** the summary span brackets the grid's daily highs (and lows) for the days it covers; no disagreement.

## Result

Record in the Jira **Test Result** field: `PASS yyyy-mm-dd @ 1.38.2` (or `FAIL …`). The script prints a ready-to-paste line.

## Notes / known follow-ups

- **Consistency boundary (by design).** The ranges are computed from the **provisional** snapshot pre-call, not the model's `final_snapshot`; they agree because the multi-day extended forecast is pass-through that the reconciler does not re-author. Step 3 is the live check on that assumption — a persistent prose-vs-grid divergence would justify moving the computation post-call (placeholder-render approach), tracked as a follow-up if it ever appears.
- The natural-prose quality of es/de phrasing is a human judgement (step 2); the scan only proves the token was adopted and no point was band-wrapped.
