# WX-207 — tier-'Safety' over-escalation (detector tier axis)

Shipped in **1.34.3**. Verified live by `WX-207-verify.sh` on top of a deterministic unit test.

## The bug

`DeterministicChangeDetector.PrecipTier` tiered a precip change **Safety** when EITHER the final OR
the prior block carried a safety-grade signal (severe / freezing / snow / ≥34 kt wind) — but it
applied the **prior**-block check for *every* direction. So an **Appearing/Strengthening** change
inherited Safety from a prior hazard the final no longer carries — a block going **Snow→Rain** makes
the *snow clearing* the safety event, not the appearing rain. `ValidateChangeSnapshotConsistency`
(correctly) requires the **final** block to carry the signal for an Appearing/Strengthening Safety
change, so it rejected the over-tiered change → `ChangeConsistencyException` → the report degraded:

> structured_report change 'ch1' is tier **'Safety' (Rain)** but no final_snapshot block in its
> window … carries a safety-grade signal …; **the tier is over-escalated**.

Same **class** as WX-204 (the detector emits a change the tautological validator rejects), on the
**tier** axis instead of the strength axis. Surfaced 2026-06-19 by the WX-204/205/206 verify run;
WX-205 was degrading these immediately (no wasted retries), and it recurred **every cycle** on a
Snow→Rain locality (Watonga, OK).

## The fix

`PrecipTier` now inherits Safety from the prior block **only for a Weakening/Clearing** change (a
removed hazard is safety news at any horizon, and the validator deliberately does not require final
backing for those). For **Appearing/Strengthening**, only the FINAL block's hazard qualifies —
matching the validator exactly, restoring the tautology on the tier axis.

Unit test: `DeterministicChangeDetectorTests.RainAppearing_OverPriorSnowBlock_IsNotSafetyTier_…_WX207`
(the appearing rain is tiered Plans, not Safety, and the change set passes the consistency validator;
the snow clearing stays Safety).

## Post-deploy verification (live)

`WX-207-verify.sh` (pinned to **1.34.3**), since the deploy boundary:

- **Failure signature** — *no* `the tier is over-escalated` rejection. Post-fix this should be **0**;
  any occurrence ⇒ **FAIL** (PrecipTier still over-tiers, or the old binary is live).
- **Gate** — no time window (deterministic fix); PASS needs ≥ 3 reconciliations since deploy. A
  `Claude reconciliation tokens` log line marks one reconciliation — it fires on success **and**
  degrade (the old `reconciled` success line missed degraded cycles, which is why the WX-204/205/206
  verify under-counted; fixed in the same PR).

**Run** (bash/WSL, after deploy): `docs/test-procedures/WX-207-verify.sh`. Same standing delegation
as the other verify scripts — run once ≥ 3 reconciliations accrue, record the Test Result, move
WX-207 to Done on PASS (watch Watonga, OK — it was the live recurring case).
