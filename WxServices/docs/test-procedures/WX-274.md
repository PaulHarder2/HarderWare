# WX-274 — Observability config baked into images: reboot-survival verification

**Change under test:** `prometheus` and `otel-collector` no longer bind-mount their
config files from the host; each config is baked into a thin derived image
(`observability/prometheus.Dockerfile`, `observability/otel-collector.Dockerfile`).
This removes the host-filesystem dependency that made a single-file bind mount fail
(exit 127) when Docker Desktop auto-starts the containers on reboot before the
Windows filesystem share is ready inside its Linux VM.

Run from `observability/` in **WSL bash** unless noted.

**Prerequisite:** `observability/.env` must exist with `GRAFANA_ADMIN_PASSWORD`
set (the stack fails fast otherwise — see `INSTALL.md`). It is gitignored; source
of truth is RoboForm.

## Part A — structural + health check (runnable any time, no reboot needed)

1. Rebuild and bring the stack up:
   `docker compose up -d --build`
   **Expect:** prometheus and otel-collector images build; all three containers
   report `Started`.

2. `docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Image}}"`
   **Expect:** three containers `Up`; prometheus on `observability-prometheus`,
   otel-collector on `observability-otel-collector` (compose auto-names the
   locally-built images `<project>-<service>`), grafana on `grafana/grafana:11.4.0`.

3. `docker inspect observability-prometheus-1 --format '{{range .Mounts}}{{.Type}} {{.Destination}}{{"\n"}}{{end}}'`
   **Expect:** exactly one line — a `volume` at `/prometheus`. **No** bind mount for
   `/etc/prometheus/prometheus.yml`. (This is the fingerprint: no host config path to
   race against on reboot.)

4. `docker inspect observability-otel-collector-1 --format '{{range .Mounts}}{{.Type}} {{.Destination}}{{"\n"}}{{end}}'`
   **Expect:** empty output — no mounts at all.

5. Health probes:
   - `curl -s -o /dev/null -w "%{http_code}\n" http://localhost:9090/-/healthy` → **Expect:** `200`.
   - `curl -s http://localhost:3000/api/health` → **Expect:** `"database":"ok"`.
   - `docker logs observability-otel-collector-1 --tail 1` → **Expect:** `Everything is ready. Begin running and processing data.`

6. Confirm the baked config is the intended one:
   `docker exec observability-prometheus-1 cat /etc/prometheus/prometheus.yml`
   **Expect:** matches `observability/prometheus.yml` in the repo.

## Part B — reboot survival (the acceptance-clinching test; needs a real reboot)

1. Reboot Windows (a natural Windows-Update reboot counts). Do **not** run any
   `docker compose` command afterward.

2. Once Docker Desktop has auto-started, `docker ps -a --format "table {{.Names}}\t{{.Status}}"`
   **Expect:** all three containers `Up` — in particular prometheus and
   otel-collector are **not** `Exited (127)`. No manual `--force-recreate` was needed.

3. Re-run the Part A step-5 health probes. **Expect:** all green.

**Test Result:** record `PASS yyyy-mm-dd` once Part B step 2 passes after a real
reboot. Part A alone is a `WAIT` (structural proof, but the reboot is the live test).
