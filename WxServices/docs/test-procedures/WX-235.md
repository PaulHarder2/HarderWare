# WX-235 — operator "Rerun QA" regenerates a fresh judge package

**Type:** Manual UI procedure (WxManager is not CI-built — WX-135) **+** functional verification (§13): the regeneration runs service-side, so the full path is only exercised against a running WxReport.Svc + DB. Run after deploy.

**Release:** 1.41.0
**Components:** WxManager (`QaRerunCoordinator`, `RerunQaButton`, Translation-QA + Vocabulary tabs), WxReport.Svc (`QaRerunWorker`, `TranslationQaRunner`), `QaRerunRequest` table, `GlobalSettings.GeminiApiKey`.

## Preconditions

- 1.41.0 deployed; **WxReport.Svc running** (it hosts the worker). `deploy-history.log` shows `WxReportSvc 1.41.0`.
- `GlobalSettings.GeminiApiKey` is set (else reruns fail by design — see step 8). On service start with no key, the log warns: *"GeminiApiKey is not set — operator 'Rerun QA' regenerations will fail…"*.
- At least one **enabled, generated** target language (e.g. `es`, `de`) with templates.

## Procedure

1. **Open WxManager → Vocabulary tab.** Pick a language (e.g. `de`).
   - **Expect:** a **"Rerun QA"** button beside *Save changes*.

2. **Edit a Phrase, then press Rerun QA (do NOT Save first).**
   - **Expect:** the grid **reloads from the DB**, discarding your unsaved edit (the screen snaps to the committed values); the grid goes **read-only**; the button shows the **in-progress** state — **right-pointing chevrons sweeping continuously** — and is disabled. The sweep does not flicker or restart while it runs.

3. **Watch for completion** (seconds to ~a minute — per scenario, the forecast reconcile that produces the report (Claude) runs, then the separate judge call evaluates it; the judge is judge-agnostic, the reconcile is not).
   - **Expect (success):** the button briefly shows **✓ Updated HH:mm**, then returns to active; the grid is editable again. The Vocabulary grid itself does **not** reload (a rerun doesn't change vocab).

4. **Mutual disable across tabs.** Press Rerun QA for `de`, then (while it runs) open the **Translation-QA tab** and select a `de` package.
   - **Expect:** that tab's **Rerun QA** button is already **in-progress/disabled** for `de` (shared state), and its vocabulary grid (with Copy→DB) is **read-only/locked**.

5. **Per-language freeing.** While `de` runs, switch the Vocabulary selector to a *different*, not-running language.
   - **Expect:** the button is **active** for that language (each language's state is independent). Switch back to `de` → in-progress again.

6. **Translation-QA guarded auto-reload.** On the **Translation-QA tab** with a `de` package selected, press **Rerun QA** and let it succeed.
   - **Expect:** when it completes, the tab **auto-loads the freshly regenerated `de` package** (newest `de.<stamp>`), so the verdicts now reflect the current DB. If you had switched to another language before it finished, it does **not** yank your view — you pick up the fresh package when you next select `de`.

7. **Stale-package resolution (the original problem).** Flag a vocab item "Wrong", fix it via the Vocabulary editor (Save) or Copy→DB, then Rerun QA.
   - **Expect:** after the rerun, the Translation-QA verdict for that item reflects the corrected DB value (no longer a stale "Wrong").

8. **Failure surface.** (Optional) Temporarily clear `GlobalSettings.GeminiApiKey`, restart WxReport.Svc, and press Rerun QA.
   - **Expect:** the run fails; the button shows a **persistent ⚠ Rerun failed** with the reason in its **tooltip** ("GlobalSettings.GeminiApiKey is not set…"); it stays re-pressable. Restore the key afterwards.

9. **Restart recovery.** Press Rerun QA, then stop WxReport.Svc mid-run; restart it.
   - **Expect:** the request is not stranded — the worker releases the claim on shutdown and re-runs it on the next start (the button returns to in-progress then completes), or (if it exceeded the 30-min sweep) shows ⚠.

## Result

Record in the Jira **Test Result** field: `PASS yyyy-mm-dd @ 1.41.0` (or `FAIL …`). Steps 1-7 are the core; 8-9 are robustness spot-checks.

## Notes

- The claim/sweep/complete DB logic is unit-tested headless (`QaRerunStoreTests`, SQLite-backed); this procedure confirms the live UI ↔ service round-trip the units can't.
- Judge-agnostic by design: the UI and `QaRerunRequest` track "a regeneration", never a specific judge; only `QaRerunWorker` names the concrete judge. A future non-Gemini judge needs no UI/schema change.
