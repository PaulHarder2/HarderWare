# WX-168 — per-language timing/claim validators (es plugin): post-deploy functional test

The plugin architecture and the es coverage are fully covered offline: the `ILanguageLexicon` split is
behavior-preserving for en (every pre-existing test passes through the new seam), and the es plugin is
proven by the `SpanishClosing_*` / `SpanishDayPart_*` tests (a caught es precip-at-a-dry-time and a caught
es madrugada/token contradiction, an ambiguous-`mañana` skip, and a legitimate es claim that passes).

What only a **deployed** run shows is the one production-only risk: does the es lexicon **false-reject
legitimate es prose** — degrading a *good* es report? The es checks are deliberately conservative (every
ambiguous word skips), so the risk is low, but a wrong es lexicon entry is exactly that failure mode. So
this is a **trustworthy-absence monitor**, not a "prove it fires" calibration (the offline suite already
proves it fires).

## Discrimination (why the count is trustworthy)

The es closing degrade shares its inner string with the en family — `asserts precipitation/storm activity
at a local time` — and is told apart **only by the `narrative 'es'` tag**; the es day-part degrade
likewise shares `…next to a {q:time} token that renders to…` with the en day-part check. So:

- **`WX-168-verify.sh` is entirely `'es'`-scoped** — it measures exactly the es timing/claim families.
- **`WX-177-verify.sh`'s WX-283 precip-at-a-dry-time count was scoped to `'en'`** in this same PR, so the
  es hits WX-168 newly introduces don't inflate that en trend.

en degrades → the WX-177/283 scripts; es degrades → this script; no cross-contamination.

## Procedure

Run after the 1.49.0 deploy, on/after a full active day of es cycles (es has ~1 recipient, so es reports
are infrequent — the window is longer and lower-signal than WX-139/177):

```powershell
bash docs/test-procedures/WX-168-verify.sh
```

*Expect:* reports flowed since the boundary (binary live) and **zero exhausted es timing/claim degrades**
(no false-reject observed). A non-zero count is **not** an automatic failure — read the surfaced lines:

- a **genuine** wrong-es-prose report the net correctly caught → the deterministic floor is working →
  close `PASS`;
- **legitimate** es prose the lexicon wrongly rejected → the `SpanishLexicon` is too aggressive → open a
  Bug and tighten it (the offending word), then re-verify.

## Result

`PASS` when reports flowed and no exhausted es degrade reflects a false-reject (zero degrades, or every
surfaced line is a genuine es error the net rightly caught). Set the ticket **Test Result** =
`PASS <date> @ v1.49.0`, then take In Test → Done. A quiet es window is trustworthy absence — the offline
suite carries the proof the net fires.
