# WX-264 — functional verification (§13)

**Change under test (1.45.3):** the narrative now labels time windows consistently with their own
`{q:time}` tokens — the 00–06 block uses the approved `DayPart1` word ("early hours"/"madrugada"),
a window crossing a local-day boundary names **both** bounding days, and the daypart word matches
the token's local hour. Delivered by prompt + validator (no schema/renderer change) plus one
additive glossary row.

This is an **LLM-behavioral** fix, so verification is observational (like the WX-244 / WX-246 prompt
changes): the deterministic sliver (the glossary row loaded) is a DB/log check; the labeling itself
is confirmed by re-running Translation-QA on a cross-boundary scenario and/or eyeballing live
reports.

**Boundary:** the WxReportSvc deploy of **1.45.3**. DB steps run in **Windows PowerShell**
(`sqlcmd -E -C`); the log-tail / `grep` steps run in **bash (WSL)** against
`/mnt/c/HarderWare/Logs/wxreport-svc.log` (UTC).

## 2. Cross-boundary + daypart labeling (the fix)

Re-run Translation-QA against a scenario whose storm window **crosses local midnight** (or wait for a
live report that does), for `en` plus one other language:

```powershell
cd C:\Code\HarderWare\WxServices
dotnet run --project src\WxReport.Tools.TranslationQa -- --lang de --judge gemini
```

Read the rendered `en` (and target) report and confirm, in the change-band and closing prose:

- **Both bounding days named** for a window that crosses midnight — e.g. "Monday evening into the
  early hours of Tuesday", **never** the tail day alone ("the early hours of Tuesday") nor a
  day-only span that drops a partial daypart.
- **Daypart word matches the time** — no "afternoon" beside a morning instant; the 00–06 block reads
  as the approved **"early hours"** (en) / "madrugada" (es) / "frühe Morgenstunden" (de), **never**
  "overnight" or "{weekday} night".
- A **day-only terminus is fine** when the whole day is implied ("Tuesday into the early hours of
  Wednesday", "Tuesday evening through Wednesday").

- **Expect:** the labels are internally consistent with the `{q:time}` clock the renderer shows; no
  "afternoon around 6:00 AM" self-contradiction and no dropped bounding day.

## 3. No over-suppression (the validator is a backstop, not a blocker)

The strengthened validator degrades a section only when it can *prove* a mislabel. Confirm reports
keep sending and the change-band isn't being routinely dropped:

```bash
grep -c 'Report cycle complete.' /mnt/c/HarderWare/Logs/wxreport-svc.log
grep -iE 'narrative .* (day-part|crosses a local-day|never names)' /mnt/c/HarderWare/Logs/wxreport-svc.log | tail
```

- **Expect:** cycles complete normally; the WX-264 rejection lines are absent or rare (an occasional
  retry that self-corrects is healthy — a *persistent* changeSummary degrade would mean the model
  isn't obeying the prompt and warrants a prompt tweak, not a rollback).
