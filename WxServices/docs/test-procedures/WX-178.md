# WX-178 — Functional test procedure

**Ticket:** WX-178 — Scheduled reports: suppress the "What's changed" band except for near-term
(local Day 1-3) severe-onset.
**Change under test:** A scheduled report's "What's changed" band is now suppressed unless a
**new severe hazard** appears in the near term — a block going `severeFlag` false→true versus
the prior committed snapshot, within **local Day 1-3** (today through the day after tomorrow).
The decision is deterministic (`ReportWorker.SuppressesScheduledChangeBand`), applied as a
post-reconcile strip (`WithoutChangeBand`); the reconciler prompt is aligned to the same rule.
Unscheduled updates are unchanged.
**Released in:** 1.31.0 · **Component:** WxReport.Svc

## Deterministic proof (the primary verification)

The decision is a pure function and is proven by unit tests in
`tests/WxReport.Tests/ScheduledChangeBandTests.cs` (run with `dotnet test WxServices.sln`):

- `Scheduled_NonSevereChange_SuppressesBand` — the founding defect / send 2285 case.
- `Scheduled_NearTermSevereOnset_KeepsBand` — a Day 1-3 not-severe→severe onset keeps the band.
- `Scheduled_SevereOnsetOnDay4_SuppressesBand` — a Day-4+ onset does **not** justify a band.
- `Scheduled_SevereAlreadyPresentInPrior_NotAnOnset_SuppressesBand` — present ≠ onset.
- `Scheduled_NoPriorSnapshot_NeverSuppresses`, `Scheduled_ReportHasNoBand_NothingToSuppress`,
  `Unscheduled_NeverSuppresses_EvenWithoutSevere`.
- `HasSevereEscalationOver_*` — the near-term cutoff overload, incl. the exact `< cutoff` boundary.

Because the logic is deterministic and snapshot-driven (no per-recipient or locale variance in
the decision), these tests are the authoritative correctness check. The live step below confirms
the **deployed binary and the wiring** behave — it is a smoke check, not the proof.

## Live spot-check (post-deploy smoke)

Report **kind is not persisted** on `CommittedSends`; it is read from the EmailBody header
("Scheduled Report"). The band renders as "What's changed:". So, after the deploy boundary:

1. List scheduled reports delivered since the deploy that **carry a band**:

   ```bash
   powershell.exe -NoProfile -Command "sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -h -1 -W -Q \
     \"SET NOCOUNT ON; SELECT Id, RecipientId, CONVERT(varchar(19),CreatedAtUtc,120) \
        FROM CommittedSends \
        WHERE IsDiagnostic=0 AND SentAtUtc IS NOT NULL AND CreatedAtUtc >= '<deploy-UTC>' \
          AND EmailBody LIKE '%Scheduled Report%' AND EmailBody LIKE '%What''s changed:%' \
        ORDER BY Id\""
   ```

   (Spanish recipients use the localized header/band labels; broaden the `LIKE`s if a
   Spanish-only locality must be covered.)

2. *Expect:* **few or none.** The fix makes a scheduled band rare — it should appear only when a
   new severe hazard has entered the next ~3 local days.

3. For any row returned, dump its body and confirm the band is justified — its forecast grid
   shows a **Severe** day within the **first ~3 rows** (local Day 1-3):

   ```bash
   powershell.exe -NoProfile -Command "sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -y 0 \
     -Q \"SET NOCOUNT ON; SELECT EmailBody FROM CommittedSends WHERE Id=<Id>\""
   ```

   *Expect:* a near-term severe day in the grid (e.g. "Severe storms likely …" in row 1-3). A
   scheduled report carrying a band for an **ordinary, non-severe** change — the 2285 regression
   (a rain→thunderstorm upgrade with severe only on a far day, or no severe at all) — must **not**
   appear.
