# WX-211 — Test Procedure

**Change:** WX-172 follow-up. Template generation no longer runs at the top of `RunCycleAsync`
(ahead of the report sends); the send phase is extracted into `RunReportSendsAsync`, and the cycle
now runs **sends → generation → count**, so a pending-language translation never delays a report —
it only extends the cycle's tail. Generation still runs on a recipient-less cycle and stays capped
at one Claude call per cycle. Plus: a generation increments `wxreport.claude.calls.total`, and the
per-call DEBUG line reads "Claude tokens" (not "Claude reconciliation tokens").

**Ships in:** WxReport.Svc (`ReportWorker`), release **1.35.1**.

**Test Proc field:** `docs/test-procedures/WX-211.md`
**Test Result field:** `PASS yyyy-mm-dd @ 1.35.1` / `FAIL …` (recorded after the §2 in-production observation).

---

## 1. Unit tests (pre-merge)

No **new** unit tests: `RunReportSendsAsync` is a behaviour-preserving extraction of the existing
send phase (same code, same order), and the cycle itself is integration-level (DB + Claude + SMTP,
no test harness). The regression guard is the full suite staying green:

```bash
dotnet test WxServices/WxServices.sln
```

**Expect:** all pass (WxReport.Tests unchanged at 453) — confirms the extraction didn't regress the
send path or the WX-172 generation/validation tests.

## 2. Functional verification (post-deploy — operator-triggered, MANUAL)

WX-211's behaviour is only observable when a language **actually generates**, which needs a fresh
enable (the already-READY `eo`/`en`/`es` won't re-generate). The WX-172 DB invariants are unchanged
by this PR, so this procedure observes only the reorder + instrumentation.

**Baseline smoke first:** 1.35.1 in `deploy-history.log` (WxReportSvc), services up, heartbeat
current, no new ERRORs since the deploy.

Then:

1. **Enable a fresh language.** In WxManager → Languages, enable one with no templates yet (e.g.
   *German* `de`).
2. **Wait one report cycle**, then in `wxreport-svc.log` find the cycle that generated `de`.
   **Expect (item A — the reorder):** within that cycle, the
   ```text
   WX-172: generating templates for 'de' (German)…
   WX-172: 'de' generated and READY — N token(s).
   ```
   lines appear **after** that cycle's `Report cycle complete.` line — i.e. the reports went out
   first, then generation ran. (Before WX-211 the generation lines preceded the sends.)
3. **Item C — log wording.** The per-call token line for the generation reads:
   ```text
   translate:de: Claude tokens [WX-172 template generation] — in=… out=… cache-read=… cache-write=…
   ```
   **Expect:** "Claude **tokens**", **not** "Claude reconciliation tokens".
4. **Item B — calls counter.** On the WxReport Grafana dashboard, the **"Claude API Calls"** panel
   (`wxreport_claude_calls_total`) ticks up by one for the generation cycle (previously a generation
   moved only the token/cost panels).
5. **(Opportunistic — item A's recipient-less property.)** If a generation happens on a cycle with
   no reports due (e.g. outside send hours, or a pending language while no report is scheduled), the
   `translate:` line still appears — generation is reached even when the send phase did nothing.

## 3. Automated check — `WX-211-verify.sh` (FINAL step)

> **Do not run this step unless the §2 observations passed** (especially: a fresh language was
> enabled and its generation lines landed *after* the cycle's "Report cycle complete"). The script
> confirms the durable log fingerprints; it does not replace the by-eye ordering check in §2.

Once §2 is clean, run it against the live log:

```bash
bash WxServices/docs/test-procedures/WX-211-verify.sh
```

It resolves the deploy boundary version-pinned (`deploy-info.sh --version 1.35.1 WxReportSvc`) and
asserts, since that boundary:

- **Zero `Claude reconciliation tokens` lines** — the retired wording (item C). This is the
  deterministic, always-present new-binary signal: the old binary logged it on every reconciliation,
  so any occurrence means the old binary is still live. (Positive counterpart: `Claude tokens [`
  lines, the new wording.)
- **Zero reorder violations** (item A) — no cycle in which a `generating templates` line preceded
  that cycle's `Report cycle complete.` line (the old top-of-cycle order).
- **≥ 2 report cycles** since deploy (PASS gate), and the background-ERROR delta vs the equal-length
  pre-deploy window.

**Expect (PASS):**

```text
                                          total : 0   (expect 0)
   ====>  PASS   WX-211 verified live: …
```

- Before deploy: `WAIT` (version not in `deploy-history.log`).
- If no generation has run since deploy, the script notes the reorder isn't yet *positively*
  exercised (the old-wording check still proves the new binary); enable a fresh language and re-run.
- Any nonzero total: `FAIL` — a retired wording line (old binary / item C not deployed) or a
  generation that ran before its cycle's send-complete (the old order).

Record the verdict's `Test Result` string (`PASS yyyy-mm-dd @ 1.35.1` / `FAIL …`) in the Jira
**Test Result** field.
