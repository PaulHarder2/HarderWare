# WX-238 — Translated report free-generates target terms instead of approved vocabulary

**Bug · verifies the approved-vocabulary glossary reaches the free-composed narrative.**

## The bug

The translation-QA judge returned vocabulary verdicts **clean**, yet the assembled report's
free-composed narrative (`changeSummary` / `closing`, WX-128) used a target term that **differs
from the approved `LanguageTemplates` token**, because that prose free-generates in the target
language and never saw the curated vocabulary:

- es `winter-frozen`: `"llovizna engelante"` where `drizzle_freezing` = approved `"llovizna helada"`.
- da `winter-frozen`: `"Isregn"` where the approved freezing-drizzle term is `"frysende støvregn"`;
  the wintry-mix term `"vinterblandingsnedslag"` went unused.

The structured body (grid, conditions, banner) is token-substituted and never drifts — only the
narrative did.

## The fix

A language-neutral **`PromptGlossaryTokens`** registry lists the weather concepts to anchor. The
reconciler injects an **approved-vocabulary glossary** into its per-report (uncached) prompt block —
for every requested narrative language **including English** — as `english concept → «approved term»`;
the cached guidance instructs the narrative to use the approved term and compose naturally around it.
Prompt-anchoring (not lower temperature — that lever is removed on Sonnet 5 / Opus 4.7+), so it
survives a model migration. Determinism where the body substitutes, prompt-anchoring for the
genuinely-free narrative.

## Preconditions / sequence (per Paul, WX-238 design)

1. **PR merged and CodeRabbit review complete.**
2. **After the CR review, before deploying the changed code:** apply the migration to the live DB —
   `dotnet ef database update --project src/MetarParser.Data/MetarParser.Data.csproj --connection "<WeatherData connection string from appsettings>"` (it is additive: creates `PromptGlossaryTokens` + seeds the 39 weather-concept tokens; no existing table touched). It also auto-applies on the next `WxReport.Svc` startup, but apply it explicitly here so the schema is ready when the new code runs.
3. Rebuild the `WxReport.Tools.TranslationQa` tool from the merged code (it loads the glossary via the same DI as the service).

## Procedure

1. Re-run the translation-QA generation + judge for **all four** languages (Windows PowerShell,
   from the solution root `C:\Code\HarderWare\WxServices`):

   ```powershell
   dotnet run --project src\WxReport.Tools.TranslationQa -- --lang es --judge gemini
   dotnet run --project src\WxReport.Tools.TranslationQa -- --lang da --judge gemini
   dotnet run --project src\WxReport.Tools.TranslationQa -- --lang de --judge gemini
   dotnet run --project src\WxReport.Tools.TranslationQa -- --lang eo --judge gemini
   ```

   Each writes a fresh run under `C:\HarderWare\translation-qa\<lang>.<stamp>\`.

2. Run the checker (WSL bash) — it inspects the newest es/da rendered target reports for the
   documented divergences:

   ```bash
   bash docs/test-procedures/WX-238-verify.sh
   ```

## Pass criteria

- **es:** the rendered `winter-frozen` report uses `"helada"` (approved) and **not** `"engelante"`.
- **da:** the rendered `winter-frozen` report uses `"frysende støvregn"` **and** `"vinterblandingsnedslag"`.
- The es/da `judged.json` `reportFindings` no longer flag these vocabulary divergences.
- de/eo re-run for regression: no **new** approved-token divergence in their `reportFindings`
  (manual read — the checker asserts the two documented es/da cases; de/eo Tier-1/Tier-3 findings are
  WX-239/WX-240, out of scope here).

Option B is prompt-anchoring — **probabilistic by design** (Paul, agreed). A single clean re-run with
the three named divergences gone is PASS. If a divergence recurs, strengthen the glossary/instruction
wording and re-run.
