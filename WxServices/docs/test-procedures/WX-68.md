# WX-68 Functional Test Procedure — reboot survival of the containerized stack (§13)

**What this verifies:** that after a Windows reboot and Docker Desktop autostart, the full stack
comes back up **without manual intervention** — all four Wx services `running` + `healthy`, the
observability stack up, and the **autoheal sidecar back unattended**. The autoheal piece is the
hard case WX-68 had to solve: autoheal's only mount is the Docker socket (a single *file*), and on
a Docker Desktop reboot the daemon can re-start the container against a stale baked
`docker-desktop-bind-mounts` path → dir-onto-file → `Exited(127)`, which `restart: unless-stopped`
**cannot** heal (a mount-*create* failure never increments RestartCount). Same failure family as
the stale-bind-mount bugs WX-274 / WX-278, but the socket cannot be baked into an image — so the
fix is an **at-logon scheduled task** (`Reconcile-AutohealSidecar.ps1`, registered by
`Register-AutohealTask.ps1`) that waits for the Docker engine and `--force-recreate`s autoheal only
if it is not already running.

**Recording results:** on pass, set the ticket's custom fields — **Test Proc** =
`docs/test-procedures/WX-68.md`, **Test Result** = `PASS yyyy-mm-dd @ <version>`. On failure:
`FAIL yyyy-mm-dd @ <version> — <which check>` plus a ticket comment.

---

## 0. Preconditions — arm the reconcile task (one-time, elevated)

The reboot fix is **not armed** until the logon task is registered. Once per machine, in an
**elevated** Windows PowerShell (triggers a UAC / secure-desktop prompt). The `tools` path below is
the **repo clone** location — adjust if your clone is not at `C:\Code\HarderWare`:

```powershell
cd C:\Code\HarderWare\WxServices\tools
.\Register-AutohealTask.ps1
```

Confirm afterward (non-elevated is fine):

```powershell
Get-ScheduledTask -TaskName 'HarderWare Autoheal Reboot Reconcile' |
  Select-Object TaskName, State
```

**Expect:** `State = Ready`. The task runs **at logon, as the interactive user, RunLevel Highest**
— interactive because Docker Desktop's engine pipe is session-bound, so a SYSTEM-at-startup task
would race the engine and never see it.

## 1. ⚠️ Back up first, then reboot

This test reboots the production box. **Before rebooting**, run the backup refresh:

```bash
bash /mnt/c/Code/tools/backup-refresh.sh
```

Then reboot Windows normally and log back in. Give Docker Desktop a minute to autostart the engine
and the compose stacks.

## 2. Confirm all containers are back — running + healthy

From WSL bash:

```bash
powershell.exe -NoProfile -Command "docker ps --format '{{.Names}}\t{{.Status}}' | Sort-Object"
```

**Expect** eight containers up, and the four services **plus autoheal** reporting `(healthy)`:

- `services-wxmonitor-1` — `Up … (healthy)`
- `services-wxparser-1` — `Up … (healthy)`
- `services-wxreport-1` — `Up … (healthy)`
- `services-wxvis-1` — `Up … (healthy)`
- `services-autoheal-1` — `Up … (healthy)`
- `observability-grafana-1` / `observability-otel-collector-1` / `observability-prometheus-1` — `Up …`

No `docker compose up` was run by hand — that is the whole point.

## 3. Confirm autoheal is genuinely healthy (not the 127 mount bug)

```bash
powershell.exe -NoProfile -Command "docker inspect services-autoheal-1 --format 'State={{.State.Status}} Health={{.State.Health.Status}} RestartCount={{.RestartCount}} ExitCode={{.State.ExitCode}}'; docker inspect services-autoheal-1 --format '{{range .Mounts}}{{.Type}} {{.Source}} -> {{.Destination}}{{println}}{{end}}'"
```

**Expect:** `State=running Health=healthy ExitCode=0`, and the mount is the real socket —
`bind /var/run/docker.sock -> /var/run/docker.sock`, **not** a `docker-desktop-bind-mounts/…`
path. (RestartCount can be 0 — see the note below.)

## 4. Confirm the reconcile task fired on logon (a FRESH entry)

`tail` alone can show an *old* success, which would not prove the logon task ran **this** reboot —
so first note the boot time, then confirm the newest reconcile entry is newer than it:

```bash
# boot time (UTC), for comparison
powershell.exe -NoProfile -Command "(Get-CimInstance Win32_OperatingSystem).LastBootUpTime.ToUniversalTime().ToString('yyyy-MM-dd HH:mm:ss')"
# The log is {InstallRoot}\Logs\autoheal-reconcile.log — the WSL path below assumes the default
# InstallRoot C:\HarderWare; adjust for a non-default install.
tail -5 /mnt/c/HarderWare/Logs/autoheal-reconcile.log
```

**Expect** the newest entry's timestamp to be **after the boot time above** (proving this reboot's
logon task ran, not a stale success), and to be one of two healthy outcomes:

- `… INFO  autoheal already running (<id>) - nothing to reconcile.` — Docker Desktop restarted
  autoheal cleanly with the correct mount, so the task correctly no-op'd. **This is a pass.**
- `… INFO  Autoheal reconcile start …` → `… INFO  autoheal not running (state: exited …) - force-recreating…` → `… INFO  autoheal reconciled and running (<id>).` — the task caught autoheal
  down and healed it. **Also a pass** — the mechanism doing its job.

Either *fresh* outcome is green. A line at level `ERROR` — the shared PowerShell log format is
`<UTC-timestamp> ERROR <message>`, **no brackets** — or autoheal still down after the task ran, is
a **FAIL**.

---

**PASS** = task armed and `Ready` (step 0); all eight containers back unattended — the four
services + autoheal `healthy` and the three observability containers `Up` (step 2); autoheal on the
real socket mount with `ExitCode=0` (step 3); the reconcile log shows a **fresh** (post-boot)
healthy outcome (step 4).

**Note on RestartCount / the two healthy paths.** Whether the reconcile task *no-ops* or *force-
recreates* on a given reboot depends on whether Docker Desktop's own `unless-stopped` restart
happened to re-mint the socket mount correctly that boot. Both are passes: the safety net is armed,
fires on logon, no-ops when autoheal is already healthy, and force-recreates when it is down. The
force-recreate path is exercised whenever autoheal comes up `Exited(127)` (stale mount) or
`Exited(137)` (killed during the Docker Desktop lifecycle).
