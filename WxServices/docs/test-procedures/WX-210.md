# WX-210 — Test Procedure

**Change:** MetarFetcher reconciles the METAR insert by correction rank — collapses within-response duplicates (the KJXI triplication that violated `UX_Metars_Station_Time_Type` and rolled back the whole batch), lets a later-arriving `COR` overwrite a stored observation in place, never lets a non-`COR` clobber a stored `COR`, and retries inserts row-by-row on a batch fault.

**Ships in:** WxParser.Svc (`MetarParser.Data.MetarFetcher`), release **1.34.4**.

**Test Proc field:** `docs/test-procedures/WX-210.md`
**Test Result field:** `PASS yyyy-mm-dd @ 1.34.4` / `FAIL …` (recorded after the §13 in-production run).

---

## 1. Unit tests (pre-merge — the correctness proof)

`MetarFetcherDedupTests` in `tests/MetarParser.Tests`. Run:

```bash
dotnet test WxServices/tests/MetarParser.Tests/MetarParser.Tests.csproj
```

**Expect:** all pass. The suite covers, directly:

- **Within-response collapse** — identical copies collapse to one; a `COR` wins over a non-`COR` in *either* feed order; distinct keys all kept; a duplicate for one station does not drop a co-batched station.
- **Reconcile rules** — new key inserts; incoming `COR` overwrites a stored non-`COR`; **a stored `COR` is not overwritten by a later non-`COR`**; an identical re-arrival skips; two differing `COR`s resolve to the later with a warning.
- **EF mechanics (SQLite in-memory)** — the `COR` overwrite is **in place** (same primary key, child sky/weather/RVR rows replaced, no orphaned child rows); the `ApplyPlan` **guard refuses** a non-`COR` overwrite of a stored `COR` even when handed a deliberately-bad plan; `InsertPerRow` lands the good row when a co-batched duplicate collides on the unique index.

> The `COR`-overwrite and guard behaviour is verified **here**, not in production: the live AWC feed issues no corrections on demand, so the functional check below can only confirm the dedup / no-crash half.

## 2. Functional verification (post-deploy — `WX-210-verify.sh`)

After 1.34.4 deploys (WxParserSvc reaches `deploy-history.log`), run:

```bash
bash WxServices/docs/test-procedures/WX-210-verify.sh
```

The script resolves the deploy boundary version-pinned (`deploy-info.sh --version 1.34.4 WxParserSvc`) and reads `wxparser-svc.log` since that boundary.

- **Failure signature:** any `UX_Metars_Station_Time_Type` duplicate-key violation since the deploy. **Expect 0.**
- **Exercised / PASS gate:** at least `MIN_CYCLES` (6) `METAR fetch done` cycles since the deploy. KJXI re-serves its duplicate every ~10-minute cycle, so a handful of clean cycles is strong evidence the within-response collapse holds.
- **Background health:** new `ERROR` lines vs the equal-length pre-deploy window (steady unrelated noise cancels).

**Expect (PASS):**

```text
UX_Metars duplicate-key violations       : 0   (expect 0 -- the failure signature)
====>  PASS   WX-210 verified live: no failure signature, change exercised.
```

- Before deploy, or before 6 cycles have logged: `WAIT` (correct — not yet conclusive).
- Any `UX_Metars_Station_Time_Type` line since the boundary: `FAIL` — the collapse is not deduping KJXI's repeat, or the old binary is still live.

Record the verdict's `Test Result` string in the Jira **Test Result** field.
