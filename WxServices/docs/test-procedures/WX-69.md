# WX-69 Functional Test Procedure — WxManager Setup + Configure tabs, post-containerization

**What this verifies:** that WxManager's **Setup tab** and **Configure tab** reflect the
post-containerization reality — the host-side wgrib2 / Conda / wxvis-package prerequisites are gone
(they live inside the WxParser/WxVis images now), Docker is a required prerequisite, no runtime
per-container health rows were added, the dead wgrib2/Conda path fields are removed from Configure,
InstallRoot is display-only, and the Configure header no longer over-promises that the services pick
up changes.

This is a **WPF UI change** — WxManager is **not** CI-built, so the verification is a manual
launch-and-eyeball against the live containerized stack. There is no automated script.

**Recording results:** on pass, set the ticket's custom fields — **Test Proc** =
`docs/test-procedures/WX-69.md`, **Test Result** = `PASS yyyy-mm-dd @ <version>`. On failure:
`FAIL yyyy-mm-dd @ <version> — <which check>` plus a ticket comment.

---

## 0. Preconditions

- The four service containers are up (from WX-68): `docker ps` shows `services-wxparser-1`,
  `-wxreport-1`, `-wxvis-1`, `-wxmonitor-1` running, plus SQL Server on the host.
- Build WxManager locally (WPF, not CI): from **Windows PowerShell**
  `cd C:\Code\HarderWare\WxServices; dotnet build src\WxManager\WxManager.csproj -c Debug` — expect
  `Build succeeded, 0 Warning(s), 0 Error(s)`.

## 1. Setup tab — three rows, Docker required

Launch WxManager. On the **Setup** tab:

- **Expect exactly three check rows**: **SQL Server**, **Database**, **Docker**.
- **Expect NO rows** for wgrib2, Conda Python, or wxvis packages (retired).
- **Expect NO per-container / network rows** (WxParser/WxReport/WxVis/WxMonitor health, `harderware`
  network) — runtime health is owned by WxMonitor + autoheal + Grafana, not this tab.
- With SQL Server, the database, and Docker Desktop all up: all three go green and the status reads
  **"All checks passed (3/3)."**, and the other tabs enable.

## 2. Setup tab — Docker is now load-bearing

Confirm the Docker check's hint no longer says "(optional …)". *(Docker-down behavior — whether the
other tabs stay disabled — is the design decision recorded on the ticket; verify it matches the
decision taken: __gate-on-Docker__ or __Docker-informational__.)*

## 3. Configure tab — Paths group is InstallRoot only, read-only

On the **Configure** tab, **Paths** group box:

- **Expect only one field: Install Root** — the **Conda Python Exe** and **wgrib2.exe Path** fields
  are gone.
- **Install Root is read-only**: clicking it does not let you type; its tooltip explains it is set in
  `appsettings.shared.json` before deploy (or via `WXSERVICES_INSTALL_ROOT`).
- The value shown matches the system's actual install root (e.g. `C:\HarderWare`).

## 4. Configure tab — honest header + Save still works for the live fields

- The header line no longer says *"Services pick up changes automatically."* It now notes that
  changes here apply to the desktop apps, and secrets go to the database.
- Edit a still-live field (e.g. **Claude Model** or an **SMTP** field), click **Save Configuration**
  → status shows **"Configuration saved."** with no error.
- Confirm `{InstallRoot}\appsettings.local.json` was written and **contains no** `InstallRoot`,
  `Gfs:Wgrib2Path`, or `WxVis:CondaPythonExe` keys (those are no longer written from this tab).
- Secrets (SMTP username/password/from, Claude API key) still round-trip via the `GlobalSettings`
  DB row (the **Test** buttons still work).

---

**PASS** = Setup tab shows the three rows with Docker required and no per-container rows (step 1–2);
Configure Paths group is InstallRoot-only and read-only (step 3); the header is honest and Save
writes local.json without the three removed keys while secrets still round-trip (step 4).
