# WX-291 Manual Test Procedure — WxManager / WxViewer window placement

**What this verifies:** both WPF apps restore the position and size they had at
last close (including maximized state), default to **1920x1080** on first run,
and — the safety guarantees — always open fully visible on a connected monitor
and never larger than the monitor they land on. The pure placement math
(`WindowPlacement.ClampToWorkArea` / `CenteredDefault` / `Load` / `Save`) is
covered by `WxServices.Common.Tests`; the WPF apps have no automated UI harness
and are not CI-built (WX-135), so this document is the verification record for
the live restore/persist behavior.

**Recording results:** when every section passes, set the ticket's custom
fields: **Test Proc** = `docs/test-procedures/WX-291.md` and **Test Result** =
`PASS yyyy-mm-dd @ <version>`. If anything fails, set
`FAIL yyyy-mm-dd @ <version> — section N` and describe it in a ticket comment.

The placement files live at:

- `%LOCALAPPDATA%\HarderWare\wxmanager.window.json`
- `%LOCALAPPDATA%\HarderWare\wxviewer.window.json`

"**Expect:**" lines are pass/fail criteria — if reality differs, note the
section and continue; the list of failed sections is the run's deliverable.

---

## 0. Setup

1. Build everything — open **PowerShell**:

   ```powershell
   cd C:\Code\HarderWare\WxServices
   dotnet build WxServices.sln
   ```

   Confirm the output ends with **Build succeeded**.
2. Deploy both apps the usual way. The title bar shows the version under test.
   (Window placement is applied at window construction, independent of the
   WxManager **Setup** tab prerequisite checks documented in INSTALL.md — the
   window opens, restores, and persists whether or not those pass, so there's no
   need to wait for the pre-flight to go green for this procedure.)
3. A convenience for the steps below — inspect the current placement file at any
   time:

   ```powershell
   Get-Content "$env:LOCALAPPDATA\HarderWare\wxmanager.window.json"   # or wxviewer.window.json
   ```

## 1. First-run default (no saved state) — 1920x1080, centered

Run this for **each** app. Set `$app` to the one under test and repeat the whole
section for the other — the commands below use `$app` so neither app is silently
skipped:

```powershell
$app = "wxmanager"   # then repeat this section with:  $app = "wxviewer"
```

1. Close the app if it is open. Delete its placement file:

   ```powershell
   Remove-Item "$env:LOCALAPPDATA\HarderWare\$app.window.json" -ErrorAction SilentlyContinue
   ```

2. Launch the app.
   **Expect:** it opens **windowed** (not maximized), sized **up to 1920x1080
   but capped to the primary monitor's work area** (the desktop minus the
   taskbar and any docked appbars), centered on that monitor. Concretely:
   - On a monitor **larger** than 1920x1080 (e.g. the 4K TV): exactly
     **1920x1080**.
   - On an **exactly 1920x1080** monitor: full width, height minus the taskbar
     — e.g. **1920x1032** with a standard ~48px bottom taskbar (not a hard 1080
     that would slide under the taskbar).
   - On a monitor **smaller** than 1920x1080: shrunk to fit that work area, still
     fully visible (AC6).

   *Units:* the figures above are **device-independent pixels** at **100% display
   scaling** (the stored JSON is in DIPs, not physical pixels). At a different
   scale factor — e.g. 150% on the 4K TV — the on-screen physical size differs
   and the JSON values won't equal raw pixel measurements; what must hold is that
   the window fits the work area and is centered, not a literal 1920x1080 count.

   For WxViewer specifically, confirm it no longer force-opens maximized (AC4).

## 2. Restore position and size — normal (windowed) state

Run for **each** app.

1. With the app open and **not** maximized, drag it to a deliberate spot (e.g.
   onto the 4K TV) and resize it to a distinctive, non-default size.
2. Close the app.
3. Inspect the file — **Expect:** `Left`/`Top`/`Width`/`Height` match roughly
   where you left it and `Maximized` is `false`.
4. Relaunch.
   **Expect:** the window reappears at the same position and size (within a few
   pixels of coordinate rounding) — AC1 / AC2.

## 3. Restore maximized state

Run for **each** app.

1. Open the app, **maximize** it (double-click the title bar or the
   maximize button), then close it while maximized.
2. Inspect the file — **Expect:** `Maximized` is `true`, and `Width`/`Height`
   hold the *underlying* (restored) size, not the full screen size — i.e. the
   `RestoreBounds`, so an un-maximize returns to a sensible window.
3. Relaunch.
   **Expect:** the app opens **maximized** on the same monitor. Un-maximize it.
   **Expect:** it restores to the pre-maximize windowed size/position.

## 4. Always visible — saved off a now-disconnected monitor (AC5)

This is the core motivation: a window last placed on the 4K TV must not strand
off-screen when the TV is gone. Choose **one** of 4a (real) or 4b (simulated).

> **Representative coverage:** sections 4 and 5 each exercise a single app
> (WxManager here for AC5, WxViewer in §5 for AC6). This is intentional — both
> apps apply placement through the *same* shared path
> (`WindowPlacementExtensions` → `WindowPlacement.ClampToWorkArea`), so a
> guarantee proven for one holds for the other. Run the other app too if you
> want belt-and-suspenders, but it exercises identical code.

**4a — real (if you can disconnect a monitor):**

1. Place WxManager on the secondary monitor / TV, close it.
2. Physically disconnect that monitor (or turn the TV off / switch the dock so
   only the 1920x1080 monitor is active).
3. Launch WxManager.
   **Expect:** it opens fully visible on the remaining monitor — never in
   off-screen null-space, and grabbable by its title bar.

**4b — simulated (single-monitor box):** with the app closed, hand-edit the file
to an off-screen rectangle, then relaunch:

```powershell
Set-Content "$env:LOCALAPPDATA\HarderWare\wxmanager.window.json" '{ "Left": 8000, "Top": 4000, "Width": 1000, "Height": 700, "Maximized": false }'
```

   **Expect:** WxManager opens fully on the current monitor (slid inside the work
   area), not at (8000, 4000).

## 5. Never oversized — saved larger than the current monitor (AC6)

With the app closed, hand-edit the file to a 4K-sized window, then relaunch on
the 1920x1080 monitor:

```powershell
Set-Content "$env:LOCALAPPDATA\HarderWare\wxviewer.window.json" '{ "Left": 0, "Top": 0, "Width": 3840, "Height": 2160, "Maximized": false }'
```

   **Expect:** WxViewer opens no larger than the 1920x1080 monitor's work area
   (shrunk to fit), fully visible.

## 6. Corrupt / missing file is harmless (AC7)

1. With the app closed, write garbage to the file:

   ```powershell
   Set-Content "$env:LOCALAPPDATA\HarderWare\wxmanager.window.json" '{ not valid json'
   ```

2. Launch WxManager.
   **Expect:** it opens at the 1920x1080 default (section 1), no crash, no error
   dialog. On the next clean close the file is rewritten as valid JSON.

---

**Scope note:** only each app's **main** window is persisted; secondary dialogs
(e.g. WxViewer's Recipients dialog) are intentionally out of scope for WX-291.
