# WX-190 — Daypart clock-band grid + 24-hour-clock legend

**Change:** the forecast grid's Conditions cell is re-architected from precipitation-episode
lines (labeled "Overnight"/"Morning"/…) to a full-day **clock-band tiling**: each local day is
partitioned into its four native 6-hour blocks, labeled on a 24-hour clock (`00-06` / `06-12` /
`12-18` / `18-24`), adjacent same-condition bands merge, and a uniform day collapses to one
`00-24` line. A one-line legend beneath the grid keys the clock. In prose with no grid row to
anchor the day (the degraded hazard banner, the WX-189 fallback band), the 00-06 pre-dawn block
reads `{weekday} 00-06` rather than a floating "overnight". Ships in `WxReportSvc`, v1.33.0.

**Invariant under test (against the deployed system, real data):** every grid-bearing report
rendered since the deploy carries the 24-hour-clock legend, and within each day the clock bands
tile contiguously (no daypart-coverage gap). No deterministic surface emits a floating
"Overnight" daypart label.

**Why a deployed-system check at all:** the rendering is deterministic and fully unit-tested, but
this is recipient-visible editorial output on a safety product, and real forecasts exercise band
merges, severe bands, and whole-day `00-24` collapses across a variety the synthetic fixtures
can't. The persisted `CommittedSends.EmailBody` (stored pre-meteogram-insertion, so the rendered
grid + legend HTML is present) is the evidence.

**Gate (no wait window — count-based, per the WX-190 design pass):** this is a deterministic
render change, so there is nothing to settle over wall-clock time. PASS is gated on a minimum of
**3 report-issuing cycles** since the deploy (a cycle = one reconciled `ForecastSnapshot` that
issued ≥1 grid-bearing report; the deploy's startup **Diagnostic** report counts as one). Until
3 cycles have shipped → WAIT. A failure signature → FAIL at any point.

- **Test Proc:** `docs/test-procedures/WX-190.md`
- **Test Result:** `PASS yyyy-mm-dd @ 1.33.0` (or `FAIL …`) — recorded after the run below.

## Automated check — `WX-190-verify.sh`

Reads the database via `sqlcmd` against `.\SQLEXPRESS` / `WeatherData`: a render change leaves no
*change-specific* service-log fingerprint, so the evidence lives in `CommittedSends.EmailBody`.
Resolves the deploy boundary version-pinned to 1.33.0 for `WxReportSvc` through `deploy-info.sh`,
then inspects every delivered report send since. (Baseline deploy health — services up, version in
`{InstallRoot}\Logs\deploy-history.log`, no new ERRORs in `wxreport-svc.log` over the post-deploy
window — is the standard post-deploy smoke and is assumed here, separate from this change-specific
check.)

It reports, and gates on:

1. **report-issuing cycles** since the deploy — distinct reconciled snapshots among grid-bearing
   sends. PASS requires ≥ 3; fewer → WAIT.
2. **grid-bearing reports missing the legend** (failure signature → FAIL) — a report whose body
   has the forecast heading but lacks the 24-hour-clock legend means the old/broken renderer ran.
3. **day-rows not covering their expected span** (failure signature → FAIL) — each *interior*
   grid day-row's clock bands must cover the whole day: start at `00`, end at `24`, contiguous in
   between (each band's end hour == the next band's start hour). An interior gap (`00-06` then
   `12-18`) is never allowed. **Two symmetric exceptions at the ends:** the *first* day-row (today)
   may **start later than `00`** — its already-elapsed dayparts are dropped (WX-195) — and the
   *final* day-row may **stop short of `24`** — the forecast horizon can truncate the last day. So
   the first row is exempt from "must start at 00", the final row from "must reach 24", and a
   single-row grid from both; every row must still be contiguous. The grid table is isolated
   (heading → legend) before this check so the hazard banner's "{weekday} 00-06" prose is never
   mistaken for a grid row. *(If an interior-day gap traces to a missing snapshot block upstream
   rather than the renderer, that is a fix-forward Bug, not a reopen — see WORKFLOW §12/§13.)*
4. **bodies containing a capitalized "Overnight"** (non-gating, manual review) — the new
   deterministic surfaces never emit one; a raw grep can't tell a grid label from a word inside
   Claude's closing prose ("overnight lows"), so this is surfaced for the step-3 confirmation
   below, not failed automatically.

### Steps

1. **Run:** `bash docs/test-procedures/WX-190-verify.sh`
   *Expect:* a header naming the deploy boundary, then the FINGERPRINT block. The verdict is
   `WAIT` until version 1.33.0 is in `deploy-history.log` and ≥ 3 report-issuing cycles have
   shipped; `PASS` once it is exercised with no failure signature; `FAIL` if any grid-bearing
   report lacks the legend or any day shows a non-contiguous band sequence.
2. **On WAIT (not enough cycles yet):** re-run after the next scheduled/unscheduled report cycle.
   The deploy diagnostic is cycle 1; two more real sends reach the threshold.
3. **Confirm any "Overnight" review lines (if listed):** open the named send's `EmailBody` and
   confirm the word sits in the closing prose, not in a grid band label or the hazard banner.
   *Expect:* every occurrence is prose; no grid/banner daypart label reads "Overnight".

## Manual visual confirmation (one real report)

1. Open the most recent **paul_en** scheduled report email (or its `EmailBody` in
   `CommittedSends`). In the Extended Forecast grid, confirm:
   - *Expect:* every Conditions cell leads each line with an `XX-YY` clock-band label
     (`00-06` … `18-24`), never "Overnight"/"Morning"/"Afternoon"/"Evening".
   - *Expect:* each interior day covers the full 00-24 — a quiet day shows a single `00-24` line; a
     mixed day shows contiguous bands whose ranges meet end-to-start and span `00`→`24`. The first
     row (today) may start later than `00` (elapsed dayparts dropped — see WX-195) and the last
     row may stop short of `24` (horizon); neither end-row is a defect.
   - *Expect:* the legend "Times use a 24-hour clock: 00 = midnight, 12 = noon, 24 = midnight."
     sits directly beneath the grid; `12 AM` / `12 PM` appear nowhere.
   - *Expect:* a severe band is bold and labeled by its clock range; if a hazard banner is
     present for a pre-dawn block it reads "{weekday} 00-06", not "overnight".
