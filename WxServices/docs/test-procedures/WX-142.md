# WX-142 Manual Test Procedure — Recipient PrecipUnit + NumberFormat

**What this verifies:** the WxManager Recipients-editor half of WX-142 — the
new **Precipitation** unit ComboBox (`in` / `mm`) loads, defaults, validates,
and persists like the existing unit controls, and the parked **`NumberFormat`**
column survives a save untouched (it has no UI and must not be clobbered to
NULL when an existing recipient is edited). The schema half (columns +
migration) is covered by `dotnet build` / `dotnet test` and the migration
apply/revert check in section 1; the WPF app has no automated UI harness and
is not CI-built (WX-135), so this document is the verification record for the
UI behavior and the standing regression for the precipitation preference.

**Recording results:** when every section passes, set the ticket's custom
fields: **Test Proc** = `docs/test-procedures/WX-142.md` and **Test Result**
= `PASS yyyy-mm-dd @ <version>`. If anything fails, set
`FAIL yyyy-mm-dd @ <version> — section N` and describe it in a ticket comment.

> ⚠️ Sections 3–5 change real data in the local `WeatherData` database. Run
> against the local dev box only, and restore the values as noted at the end
> of each section. "**Expect:**" lines are pass/fail criteria — if reality
> differs, stop, note the section, and record a FAIL.

## 2. Precipitation control — display, options, default

1. Click the **Recipients** tab; select an existing recipient.
   **Expect:** below **Wind Speed** there is a **Precipitation** row with a
   ComboBox. Its value matches the recipient's stored `PrecipUnit`
   (`in` for everyone, post-migration).
2. Open the Precipitation dropdown.
   **Expect:** exactly two options, **in** and **mm**, in that order.
3. Click **New**.
   **Expect:** the form clears to defaults and Precipitation shows **in**
   (alongside Temperature **F**, Pressure **inHg**, Wind Speed **mph**).

## 3. Dirty-tracking + persistence

1. Select an existing recipient (note their current Precipitation value).
2. Change Precipitation from **in** to **mm**.
   **Expect:** the **Save** button becomes enabled (the change is tracked).
3. Click **Save**. Re-select another recipient, then re-select this one.
   **Expect:** Precipitation now reads **mm** (persisted and reloaded).
4. Verify in the DB:

   ```powershell
   sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -Q "SELECT RecipientId, PrecipUnit FROM Recipients WHERE RecipientId='<id>'"
   ```

   **Expect:** `PrecipUnit = 'mm'`.
5. **Restore:** set it back to **in**, Save.

## 4. NumberFormat is not clobbered by an edit (the key invariant)

`NumberFormat` has no UI; editing a recipient in WxManager must leave it
untouched.

> Replace `<id>` below with a **real** `RecipientId` — list the available IDs
> first with
> `sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -Q "SELECT RecipientId FROM Recipients"`
> and copy an exact value; a mistyped id silently updates **0 rows**.

1. Seed a value directly in the DB for one recipient:

   ```powershell
   sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -Q "UPDATE Recipients SET NumberFormat='es-US' WHERE RecipientId='<id>'"
   ```

2. In WxManager, select that recipient, change an unrelated field (e.g. edit
   the **Name**, or toggle **Precipitation**), and click **Save**.
3. Re-check the DB:

   ```powershell
   sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -Q "SELECT RecipientId, NumberFormat FROM Recipients WHERE RecipientId='<id>'"
   ```

   **Expect:** `NumberFormat` is still **es-US** — the save did not null it.
4. **Restore:** set it back to NULL and undo any unrelated edit:

   ```powershell
   sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -Q "UPDATE Recipients SET NumberFormat=NULL WHERE RecipientId='<id>'"
   ```

## 5. New recipient round-trip

1. Click **New**, fill the required fields, leave Precipitation at the default
   **in**, and Save.
   **Expect:** the recipient is created with `PrecipUnit = 'in'` and
   `NumberFormat = NULL` (verify in the DB).
2. **Restore:** delete the test recipient.
