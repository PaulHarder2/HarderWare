# WX-279 — Grafana zero/low-dashboard watcher: post-deploy functional test

The watcher's detection logic (debounce, floor comparison, probe-failure handling, response parsing) is
fully covered by `GrafanaDashboardWatcherTests`. What only a **deployed** run can prove is that the real
instrument fires end-to-end: the Viewer service-account token authenticates, `GET /api/search?type=dash-db`
returns a real count, the two-cycle debounce holds, and an alert email actually sends. Like the WX-273
calibration, this is *"prove the smoke detector can go off, then trust its silence."*

The check-cadence is `Monitor:IntervalMinutes` (default 5 min); the alert needs `AlertThreshold` = **2**
consecutive sub-floor readings, so a fired alert takes ~2 cycles (~10 min).

## Setup — activate the watcher (do once)

The watcher is dormant until a grafana URL is configured. In the host-only
`{InstallRoot}\appsettings.local.json` (never the committed `appsettings.shared.json`), add under
`Monitor`:

```jsonc
"Monitor": {
  "GrafanaUrl": "http://localhost:3000",
  "GrafanaServiceAccountToken": "<the Viewer SA token from RoboForm — wxmonitor-dashboard-probe>"
  // GrafanaDashboardFloor defaults to 4 (from appsettings.shared.json)
}
```

## 1. Healthy baseline — no alert (AC#2)

1. With the four dashboards provisioned normally, restart `WxMonitor.Svc`; wait one cycle.
   *Expect:* `wxmonitor-svc.log` shows the cycle running with **no** "serving … below the floor" WARN;
   **no** alert email; `{InstallRoot}\...\publish\wxmonitor-state.json` has `GrafanaSubFloorStreak` at 0
   (or absent). A healthy grafana (4 ≥ floor 4) is quiet, including across the restart/boot window.

## 2. Calibration — force a shortfall against live grafana (AC#1)

This exercises the whole deployed path **non-destructively** by raising the floor above the real count,
rather than tearing down grafana's provisioning.

2. In `appsettings.local.json` set `Monitor:GrafanaDashboardFloor` to **5** (one above the real 4). Restart
   `WxMonitor.Svc` (or wait for the config `reloadOnChange`), then wait one cycle.
   *Expect:* a WARN like `Grafana is serving 4 dashboard(s), below the floor of 5 (consecutive reading 1 of
   2).` and **no** email yet — the debounce is holding.
3. Wait one more cycle.
   *Expect:* `... (consecutive reading 2 of 2).` and an alert email **"[WxMonitor] Grafana is serving too
   few dashboards"** whose body reports `Dashboards currently served: 4` and `Expected floor: 5`. **This is
   the calibration PASS** — real token auth + `/api/search` + debounce + email all work in production.
4. Restore `Monitor:GrafanaDashboardFloor` to **4**; wait one cycle.
   *Expect:* healthy again — `GrafanaSubFloorStreak` resets to 0, no further emails (the cooldown plus the
   now-healthy reading).

## 3. A down grafana does not false-alert (optional, proves the null path)

5. Stop grafana: `docker compose -f observability/docker-compose.yml stop grafana`. Wait two cycles.
   *Expect:* a WARN `Grafana dashboard probe failed: ...` (or `HTTP ...`) each cycle, and **no** alert email
   — an unreachable grafana is an inconclusive reading, deliberately *not* a confirmed zero. Restart it:
   `docker compose -f observability/docker-compose.yml start grafana`.

## AC#3 — threshold + cadence documented

Floor (default 4), cadence (`Monitor:IntervalMinutes`), and the 2-reading debounce are documented in
`MonitorConfig`'s XML docs, in `appsettings.shared.json`, and in this procedure — alongside the other
WxMonitor watchers.
