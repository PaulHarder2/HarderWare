# WX-290 Functional Test Procedure — canonical service token / WX-106 heartbeat blind spot (post-deployment verification)

**What this verifies:** that WX-290 (≥ 1.50.2) is live and that WxMonitor now **finds the WxParser/WxReport heartbeats** — i.e. WX-106 is resolved. Before WX-290, the writer and the monitor derived different heartbeat filenames from unrelated tokens (writer `wxparser` → `wxparser-heartbeat.txt`; monitor `wxparser-svc` → `wxparser-svc-heartbeat.txt`), so the monitor was blind and logged `heartbeat file not found` **every cycle** for both services (27k+ times pre-fix). WX-290 routes both sides through one canonical token (`WxServiceToken`), so the monitor reads exactly what the writers stamp. The success state is *silence*: zero `heartbeat file not found` WARNs while the monitor keeps cycling.

**Recording results:** on pass, set the ticket's custom fields — **Test Proc** = `docs/test-procedures/WX-290.md`, **Test Result** = `PASS yyyy-mm-dd @ <version>`. On failure: `FAIL yyyy-mm-dd @ <version> — <which service still missing>`, a ticket comment, and open the Bug.

---

## 0. Run the verification

Bash (WSL), from the repo root — `WX-290-verify.sh`, beside this procedure, reads the **WxMonitor** service log and `deploy-history.log`:

```bash
bash WxServices/docs/test-procedures/WX-290-verify.sh
```

From **PowerShell**: `wsl bash WxServices/docs/test-procedures/WX-290-verify.sh`. No arguments: the script is version-pinned to `1.50.2` / `WxMonitorSvc` and reads `wxmonitor-svc.log`. WAIT until `1.50.2` is deployed and ~1h of cycles has accrued (the fingerprint fired every cycle pre-fix, so ~12 clean cycles is already conclusive — a full active day is not needed).

## 1. Confirm the blind spot is gone (the PASS/FAIL signal)

Run the script (step 0) after ≥ ~1h past the `1.50.2` boundary.

- **Expect:** *Monitor cycles completed since deploy* > 0 (the monitor is running, so the absence is trustworthy, not idle).
- **Expect:** *"heartbeat file not found" WARNs* = **0** — the PASS signal. Pre-fix this fired ~2× per 5-minute cycle (WxParser + WxReport); post-fix the monitor reads the real files.

## 2. If a "heartbeat file not found" WARN still fired — adjudicate it

The script prints the flagged lines. Read the filename in each:

- **Old `...-svc-heartbeat.txt` name** (e.g. `wxparser-svc-heartbeat.txt`): the canonical-token fix didn't take for that service → **FAIL**; open a Bug (the writer/monitor still diverge).
- **A new/unregistered watched entry** (a service Name not in `WxServiceToken.FromConfigName`): its token fell to the normalized fallback and doesn't match the writer → **FAIL**; register the name.
- **A genuinely stale/absent heartbeat** would instead log `heartbeat is N minute(s) old` (a different string this check ignores) or a real "not found" because that service is actually down — check the service, not WX-290.

## 3. Spot-check the stale-alert path still works (non-blocking)

WX-290 changed only *which file* the monitor reads, not the stale-age logic. Optionally confirm the alert path is intact: the monitor's heartbeat watcher still emits `heartbeat is N minute(s) old` + an email finding when a heartbeat genuinely ages out (AC — the fix must not silence real alerts). Not required for PASS.

---

**PASS** = the monitor completed cycles with zero `heartbeat file not found` WARNs over ≥ ~1h past the boundary (steps 1–2). The verify script's WAIT / PASS / FAIL verdict encodes it. Then set Test Result `PASS <date> @ v1.50.2` → In Test → Done, and **WX-106 can be closed** (resolved by this).
