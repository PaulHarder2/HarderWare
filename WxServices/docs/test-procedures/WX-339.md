# WX-339 — Manual test procedure

**Ticket:** WX-339 — WxManager Vocabulary tab: display the `ValidatorUse` flag (read-only).
**Release:** v1.61.0 · **Component:** WxManager (WPF — not CI-built, so verified interactively per WORKFLOW §7a).

**What changed:** the Vocabulary tab's DataGrid gains a **read-only** "Validator" checkbox column, checked
when the entry's `LanguageTemplates.ValidatorUse` is `Yes` (the `DayPart1–4` day-part words the surviving
`{q:time}`↔day-part check keys on). Display-only — no editing, no new save path.

**Setup:** build + launch WxManager against the live `WeatherData` DB (`dotnet build WxServices.sln`, run
`WxManager`).

**The invariant under test is that each UI checkbox matches the row's `ValidatorUse` in the DB** — *not*
any specific language's flags, which are mutable (WX-338 will auto-vet de/da/eo/sq, flipping some to `Yes`).
So read current ground truth at test time rather than trusting a baked-in list:

```sql
SELECT l.IsoCode, t.Token, t.ValidatorUse
FROM LanguageTemplates t JOIN Languages l ON l.Id = t.LanguageId
WHERE t.Token LIKE 'DayPart%' AND l.IsEnabled = 1 ORDER BY l.IsoCode, t.Token;
```

**v1.61.0 snapshot** (the WX-335 load, before WX-338 runs — the concrete values the steps below name):
**es** `DayPart1`=`madrugada` is `Yes`, its other DayParts `No`; **de/eo/da/sq** all four DayPart flags
`No`. If the query shows different flags (WX-338 has since run), verify the UI against *those* instead —
the steps' es/de specifics are illustrative of "a flagged language" vs "an unflagged one", not fixed.

Read one step at a time; report the result of each. Note any failure and continue; the failed-step list is
the deliverable.

## Steps

1. Launch WxManager; open the **Vocabulary** tab.
   *Expect:* the grid loads for the default language; a new **"Validator"** column appears to the right of
   the **Repr.** column, showing a checkbox per row.

2. In the **Language** selector, choose **Spanish (es)**. Scroll to the `DayPart1` token row.
   *Expect:* the **Validator** checkbox on the `DayPart1` row (phrase `madrugada`) is **checked**; the
   `DayPart2`/`DayPart3`/`DayPart4` rows' Validator checkboxes are **unchecked**.

3. Still on es, scan the non-DayPart rows (e.g. any precipitation/label token).
   *Expect:* their Validator checkboxes are **unchecked** (only DayPart words are flagged today).

4. Select **German (de)** (or eo/da/sq). Look at the four `DayPart` rows.
   *Expect:* all four Validator checkboxes are **unchecked** (de is not yet vetted — WX-338 will set these).

5. Try to **click/toggle** a Validator checkbox (on any row, any language).
   *Expect:* it does **not** change state — the column is read-only. No cell enters edit mode; the Save
   button does not become enabled from this action.

6. Confirm **no regression** to existing editing — **non-destructively, do NOT Save** (keeps the live DB
   untouched): click into a **Note** cell (Note has no `{n}`-placeholder validation, so any change is
   valid and reliably enables Save) and append a word; then **revert** the cell to its original value.
   *Expect:* Save **enables** on the change and **disables** again on the revert; the Validator column is
   unaffected throughout. (WX-339 didn't touch the save path, so the enable/disable wiring is the
   regression signal — no persisted write is needed. Leave without saving.)

7. Confirm the footer hint text mentions Validator as read-only.
   *Expect:* the italic hint lists Validator among the read-only columns (with the "whether the day-part
   check uses this word — set by seeding and automated vetting" gloss).

## Verdict

- **PASS** — the Validator column matches each entry's DB `ValidatorUse` (v1.61.0 snapshot: es `madrugada`
  checked, de all unchecked), is not editable, and existing editing is unaffected (Save enables on an edit
  and disables on revert; no persisted write). Record `PASS yyyy-mm-dd @ v1.61.0` in **Test Result**.
- **FAIL** — record the failed step(s) and the observed behavior.
