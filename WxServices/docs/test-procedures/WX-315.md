# WX-315 Test Procedure — Configure-tab rework (DB-backed, operational-only)

**What this verifies:** that WxManager's Configure tab writes **only** to the database (operational
settings to the `Config` table, secrets to `GlobalSettings`), that the fields it no longer owns are
visibly read-only, and that the header no longer promises file-based behaviour that has not been
true since WX-313/WX-314.

**Run it interactively** — read one step, perform it, report the result; note failures and continue.
The list of failed steps at the end is the run's deliverable (WORKFLOW §7a point 4).

**Recording results:** **Test Proc** = `docs/test-procedures/WX-315.md`; **Test Result** =
`PASS yyyy-mm-dd` (or `FAIL yyyy-mm-dd — <what>`).

---

## 0. Unit tests (CI)

`dotnet test WxServices.CI.slnf` — **Expect:** all green, including
`MetarParser.Tests.OperationalConfigTests` (the tab's key mapping and port validation, which live in
the data layer precisely because WxManager itself is excluded from CI — WX-135).

## 1. The tab tells the truth

Launch WxManager and open **Configure**.

- *Expect:* the header reads *"Settings on this tab are saved to the database and are read by all
  services and desktop apps. Greyed fields are set by the setup script…"* — and says **nothing**
  about `appsettings.local.json` or services picking changes up automatically.
- *Expect:* these group headers carry their reason: **Home Location and Fetch Region** *(set by the
  setup script — re-run it to change)*, **Database** *(native / management connection — the services
  use their own)*, **Map Rendering** *(set by the setup script)*.

## 2. Read-only fields are genuinely read-only

Try to type into: Home ICAO, Home Latitude/Longitude, Bounding Box, the four Region bounds, Map
Extent, Connection String, Claude Model, Install Root.

- *Expect:* no character is accepted in any of them; each renders in muted grey; **Save stays
  disabled** (they are not dirty-tracked, so they cannot enable a save the tab could not persist).

**Read-only must still mean selectable.** Click into **Connection String**, press `End`, and select
its full value.

- *Expect:* the text cursor appears, the box scrolls horizontally to reveal the whole value, the text can
  be selected and copied with `Ctrl+C`, and typing still changes nothing. (A read-only field that
  cannot be focused would hide the tail of any value longer than its box — the connection string is
  exactly that, and this screen exists to let you read it.)

## 3. Editing an operational value enables Save

Change **SMTP Host** to a different value (e.g. `smtp.example.com`).

- *Expect:* Save becomes enabled.

## 4. Save writes to the database

Note the current time, then press **Save**.

- *Expect:* status reads *"Configuration saved."*
- Verify the row. Do **not** assume the instance or database: derive them from the connection
  string the app is actually using, shown read-only on the Database panel of the tab itself. Using
  a hardcoded `.\SQLEXPRESS` / `WeatherData` can query a different database than the one WxManager
  just wrote to, and would then "fail" against a perfectly good save (**Windows PowerShell**):

  ```powershell
  # Read Server= and Database= off the Configure tab's connection string, then:
  sqlcmd -S "<Server from the tab>" -E -C -d "<Database from the tab>" -Q "SELECT [Key],[Value],[UpdatedUtc] FROM dbo.Config WHERE [Key] IN ('Smtp:Host','Smtp:Port','Monitor:AlertEmail');"

  ```

  - *Expect:* `Smtp:Host` holds the new value and its `UpdatedUtc` is the time you just noted.

## 5. Save does NOT write a configuration file

Resolve the install root **the way the app does** — `WxPaths.ReadInstallRoot()` checks the
`WXSERVICES_INSTALL_ROOT` environment variable first, then `InstallRoot` in the
`appsettings.shared.json` sitting beside the binary, then falls back to `C:\HarderWare`. Checking
only the env var and the default would stat the wrong file on a box configured through
`shared.json`, and the assertion would then pass no matter what the app did.

```powershell
$root = $env:WXSERVICES_INSTALL_ROOT
if (-not $root) {
  $shared = Join-Path (Split-Path (Get-Process WxManager -ErrorAction SilentlyContinue).Path) 'appsettings.shared.json'
  if (Test-Path $shared) { $root = (Get-Content $shared -Raw | ConvertFrom-Json).InstallRoot }
}
if (-not $root) { $root = 'C:\HarderWare' }
$local = Join-Path $root 'appsettings.local.json'
$local; (Get-Item $local).LastWriteTime

```

Record the timestamp, perform a save, then run the last line again.

- *Expect:* the path printed is the file WxManager actually uses, and its `LastWriteTime` is
  **unchanged** across the save. The tab no longer writes the file at all; the connection string it
  contains is owned by the setup script.

## 6. The reload makes the new value live immediately

Immediately after the step-4 save, without restarting WxManager:

- *Expect:* the SMTP Host box shows the value **as persisted** (the tab re-reads from the refreshed
  configuration rather than echoing what was typed), and Save has returned to disabled.

## 7. A bad SMTP port is rejected, loudly

Set **SMTP Port** to `58y` and press Save.

- *Expect:* the save **fails** with a message naming the invalid port; the status line shows the
  failure. Re-run the step-4 query — *Expect:* `Smtp:Port` is **unchanged** in the database.
- *(Before WX-315 this silently wrote 587 and reported success — a typo that appeared to save and
  changed nothing. The loud failure is the intended new behaviour.)*

## 8. Secrets still round-trip

Change **SMTP Username** (or re-enter the password), Save, close WxManager, reopen the Configure tab.

- *Expect:* the value persisted, and **Save neither creates nor updates any secret file** — the
  secrets this tab manages (SMTP username/password/from-address, the Claude API key) live in the
  `GlobalSettings` row.

  This is a claim about *what Save writes*, not about the whole system: `What3Words:ApiKey` is still
  file-based by design (`DESIGN.md` §4.7), and `appsettings.local.json` may retain legacy fallback
  values. Those are migrated or removed separately — the What3Words move is **WX-322**, and the
  local-file prune is step 10b below. Do not read this step as "no secret exists in any file".

  > **Superseded 2026-07-21 (recorded, not rewritten).** The paragraph above was accurate when this
  > procedure was executed at v1.57.0. **WX-322 has since shipped** (v1.58.0): `What3Words:ApiKey`
  > now lives in `GlobalSettings` like every other secret, and no secret remains in any file except
  > the connection string. Re-running this procedure today should expect the database-only
  > behaviour. Left in place so the original run's reasoning still reads correctly.

## 9. Re-saving unchanged values changes nothing

Press Save again without editing anything (edit a field and revert it if Save is disabled).

- *Expect:* success, and the log directory (`Join-Path $root 'Logs'`, using `$root` from step 5 — not a hardcoded `C:\HarderWare`) shows a save logged as `0 inserted / 0 updated /
  3 unchanged`; the `UpdatedUtc` values from step 4 are **not** bumped (the timestamp records the
  last real write).

## 10. One-time migration of pre-existing values (documented manual step)

The three keys this tab now owns move from JSON to the `Config` table. There is deliberately **no**
automatic migration: only three keys move, and a startup write-path in WxManager would be a
disproportionate mechanism for a one-time move (agreed with Paul 2026-07-20).

### 10a. Carry the values into the database

On an existing box, once after deploying: open Configure, confirm SMTP Host, SMTP Port, and Alert
Email show the values you expect (they will still resolve from the JSON files until a `Config` row
exists), then press **Save** to write all three into the database.

- *Expect:* the step-4 query returns all three keys.
- *Expect:* the values in effect afterwards are the ones that were in effect before — nothing lost.

### 10b. Remove ONLY the keys that moved to `Config`

The file still holds `Smtp:Host`, `Smtp:Port` and `Monitor:AlertEmail`, which step 10a just carried
into `Config`. Once a key exists in `Config` the database wins, so those three file entries are
stale values that disagree with what is in force — and the file is the first place a human looks.

**Remove only those three.** An earlier draft of this step reduced the file to `ConnectionStrings`
alone. That was wrong and destructive — everything else in the file is either bootstrap-critical or
still file-sourced, and deleting it removes it outright:

| Key | Why it must stay |
|---|---|
| `ConnectionStrings:` | Bootstrap-critical — how the database is reached at all. |
| `Database:StartupRetry:` | Bootstrap-critical — consumed before the DB overlay loads; the provider **never** carries it, so the file is its only home. |
| `Telemetry:` | Bootstrap-critical — wired at host-build time, before the overlay. |
| `Claude:TimeoutSeconds` | Bootstrap-critical (exact-match) — same reason. |
| `What3Words:ApiKey` | A **live API key**, file-based by design until WX-322 — **which has since shipped (v1.58.0)**, moving it to `GlobalSettings`. On a current box this row no longer applies; on a pre-1.58.0 box, deleting it destroys the key. |
| `Fetch:*`, `WxVis:MapExtent`, `Claude:Model` | On a box where the setup console has **never run**, these exist only in this file — `Config` has no rows for them. Deleting them leaves the services with no home location. They are shown read-only on the tab, which is not the same as being stored elsewhere. |

Using `$root` from step 5:

```powershell
$local = Join-Path $root 'appsettings.local.json'
Copy-Item $local "$local.pre-wx315.bak"        # rollback
$json  = Get-Content $local -Raw | ConvertFrom-Json

# Remove only the three keys step 10a carried into Config.
if ($json.Smtp)    { $json.Smtp.PSObject.Properties.Remove('Host'); $json.Smtp.PSObject.Properties.Remove('Port') }
if ($json.Monitor) { $json.Monitor.PSObject.Properties.Remove('AlertEmail') }
# Drop the parent objects only if they are now empty.
if ($json.Smtp    -and -not $json.Smtp.PSObject.Properties.Name)    { $json.PSObject.Properties.Remove('Smtp') }
if ($json.Monitor -and -not $json.Monitor.PSObject.Properties.Name) { $json.PSObject.Properties.Remove('Monitor') }

$json | ConvertTo-Json -Depth 10 | Set-Content $local -Encoding UTF8
Get-Content $local
```

- *Expect:* `Smtp:Host`, `Smtp:Port` and `Monitor:AlertEmail` are gone; **everything else is
  untouched** — connection string, `Database:StartupRetry`, `Telemetry`, `Claude:TimeoutSeconds`,
  `What3Words:ApiKey`, and any `Fetch:*` / `WxVis:MapExtent` / `Claude:Model` values.
- **Then restart WxManager and re-check the Configure tab** — *Expect:* every value shows what it
  showed before. If any field goes blank, restore `.pre-wx315.bak` and stop.
**`appsettings.shared.json` is deliberately NOT pruned.** It also contains `Smtp:Host`,
`Smtp:Port` and `Monitor:AlertEmail` (and `Fetch:*`, `Claude:Model`), but it is the **generic
default layer**, it is **git-tracked**, and it ships with the repo for every install. `Config` is
layered last and wins over it (WX-313), so those entries are not stale — they are the fallback
doing its job, and WX-314's AC-6 requires leaving them intact. Editing it on one box would dirty a
tracked file and either impose local values on every install or fight the next pull. Only the
per-environment `appsettings.local.json` is pruned.

- **Also confirm a `///word.word.word` address still resolves** on the Recipients tab — that proves
  the What3Words key survived, which is the one deletion a blank Configure field would not reveal.

**PASS** = steps 0-10 meet their Expects. Then set **Test Result** `PASS <date>` → Done.
