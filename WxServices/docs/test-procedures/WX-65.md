# WX-65 Manual Test Procedure — Containerized WxVis.Svc

**What this verifies:** `WxVis.Svc` runs correctly as a Linux Docker container
(`services/docker-compose.yml`, service `wxvis`) instead of a Windows service — it builds
(with the Python + matplotlib/cartopy render stack and a Linux ODBC driver), reaches the host
SQL Server, starts, and its map workers **render** the synoptic/forecast/meteogram PNGs into the
`plots` bind mount where WxManager/WxViewer/WxReport read them. The render landing in the mount is
the end-to-end proof of three things at once: the **InstallRoot single-source-of-truth** fix
(`Program.cs` `AddInstallRoot` → the workers resolve `ScriptDir`/`OutputDir` to `/opt/wxservices`,
not the baked-in `C:\HarderWare`); the **Python SQL-auth DB path** (`db.py` +
`WxVisConfig.BuildPythonEnv` — a Linux container has no Windows identity, so the scripts
authenticate with the injected SQL login instead of `Trusted_Connection`); and the **pre-baked
cartopy Natural Earth data** (no runtime download). Note: unlike WxReport, **WxVis writes no
heartbeat** and is not one of WxMonitor's watched services, so there is no heartbeat check here.

**Recording results:** when every section passes, set the ticket's custom fields:
**Test Proc** = `docs/test-procedures/WX-65.md` and **Test Result** = `PASS yyyy-mm-dd @ <version>`.
If anything fails, set `FAIL yyyy-mm-dd @ <version> — section N` and describe it in a ticket comment.

"**Expect:**" lines are pass/fail criteria — if reality differs, note the section and continue;
the list of failed sections is the run's deliverable.

---

## 0. Setup / precondition

```powershell
$RepoRoot    = 'C:\Code\HarderWare'
$InstallRoot = (Get-Content "$RepoRoot\WxServices\appsettings.shared.json" -Raw | ConvertFrom-Json).InstallRoot
# Point the compose bind-mounts at the resolved host InstallRoot (forward slashes for Docker),
# exactly as Deploy-WxService.ps1 does. Without this a non-default InstallRoot sends the logs/plots
# to C:/HarderWare and section 3 fails.
$env:WX_HOST_LOGS_DIR  = (Join-Path $InstallRoot 'Logs')  -replace '\\', '/'
$env:WX_HOST_PLOTS_DIR = (Join-Path $InstallRoot 'plots') -replace '\\', '/'

docker info | Select-String 'Server Version'
Test-Path "$RepoRoot\services\wxvis\appsettings.local.json"
```

**Expect:** a server version prints; the path test returns `True`. (Copy
`appsettings.local.json.example` → `appsettings.local.json` and fill the SQL password from
RoboForm. The container overrides carry the DB connection string with the SQL login, the OTel
endpoint, and `WxVis:CondaPythonExe` → `/usr/bin/python3`.) Because renders are cleanly
attributable only when the native service is not also producing, ensure the native `WxVisSvc` is
**stopped + disabled** first — the deploy script's cutover does this automatically, or do it by
hand from an elevated shell.

## 1. Build + start the container

```powershell
cd "$RepoRoot\services"
docker compose up -d --build wxvis
$cid = docker compose ps -q wxvis   # this run's container ID - scope every log check below to it
docker compose ps
```

**Expect:** the image builds (Python + cartopy stack via pip wheels, `msodbcsql17`, and the
build-time cartopy 50m pre-bake) and the container starts with no error; `docker compose ps`
shows `wxvis` as `running`/`Up`.

## 2. "Application started" banner + prerequisites (reached the DB, found Python)

```powershell
docker logs $cid | Select-String 'starting\.','Database ready','Prerequisite OK','Application started'
```

**Expect:** the log shows `WxVis.Svc 1.53.0 starting.`, `Database ready.` (`EnsureSchemaAsync`
reached `host.docker.internal,1433`), `Prerequisite OK: SQL Server …`, `Prerequisite OK: Conda
Python — /usr/bin/python3` (the in-container interpreter — the `CondaPythonExe` override took),
`Prerequisite OK: wxvis packages — OK` (Python imported the matplotlib/cartopy stack in-container),
and `Application started.`.

## 3. Render lands in the plots mount — the InstallRoot + Python-DB proof

Watch the service log (written through the `Logs` bind mount) for a render, then confirm the PNG
is on the host:

```powershell
# $InstallRoot and the run's container id $cid were set in sections 0 and 1.
Get-Content "$InstallRoot\Logs\wxvis-svc.log" -Tail 20 | Select-String 'Saved forecast map','Loaded .* grid points','Loaded .* METAR'
Get-ChildItem "$InstallRoot\plots\*.png" | Sort-Object LastWriteTime -Descending | Select-Object -First 3 Name, LastWriteTimeUtc
```

**Expect:** within ~`WxVis.ForecastPollIntervalSeconds` (30s) of an un-rendered forecast hour
being available, the log shows a render that `Saved forecast map -> /opt/wxservices/plots/...png`
(the **Linux** `/opt/wxservices/plots` path proves the container authored it and the `AddInstallRoot`
seam resolved), and the loads `Loaded 27685 grid points for f…` + `Loaded ~500 METAR station
locations` (proving the **Python SQL-auth DB path** connected and queried `GfsGrid`/`Metars`). A
fresh PNG (recent `LastWriteTimeUtc`) appears in `{InstallRoot}\plots\`. **If the render log shows
a `C:\HarderWare\plots` path or a DB/login error, section 3 FAILS** (the seam or the SQL-auth path
regressed).

## 4. cartopy Natural Earth data is pre-baked (no runtime download)

```powershell
docker logs $cid | Select-String 'DownloadWarning','naturalearth'
```

**Expect:** **no** `DownloadWarning`/`naturalearth` lines at runtime — the 50m shapefiles were
baked into the image at build time, so the first render needs no S3 download and can't fail when
the host is offline after a recreate.

## 5. Telemetry to the collector / Grafana

Prerequisite: `Telemetry:Enabled=true` and the container's `Telemetry:OtlpEndpoint` ends in
`/v1/metrics`. Open Grafana (`http://localhost:3000`), or query Prometheus:

```powershell
curl.exe -s 'http://localhost:9090/api/v1/label/__name__/values' | Select-String 'wxvis'
```

**Expect:** `wxvis_forecast_renders_total` / `wxvis_render_duration_seconds_*` (and friends)
are present — the `wxvis` container reaches the collector at `host.docker.internal:4318`.

## 6. Full deploy + status summary

```powershell
# elevated - the deploy script lives in WxServices\, not services\
cd C:\Code\HarderWare\WxServices
.\Deploy-WxService.ps1 WxVisSvc     # single service first
.\Deploy-WxService.ps1 all          # then the whole stack
```

**Expect:** `WxVisSvc` alone reports the container reached `Application started`; the `all` deploy
ends with the consolidated summary showing `[OK]` for all six components (WxParserSvc service
Running; WxVisSvc/WxReportSvc/WxMonitorSvc containers Application started; WxManager/WxViewer
published). The cutover leaves the native `WxVisSvc` stopped + disabled.

## Cleanup / cutover note

If the run **passes**, the container stays up as WxVis (the native `WxVisSvc` remains disabled — a
reversible fallback; `sc delete` later once proven). If the run **fails**, take the container down
and investigate before redeploying:

```powershell
cd C:\Code\HarderWare\services; docker compose rm -sf wxvis   # 'down' takes no service name; rm -sf stops + removes just wxvis
```
