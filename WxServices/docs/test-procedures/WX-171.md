# WX-171 тАФ renderer rewire no-regression procedure + intended-delta inventory

The renderer rewire (Step 5: read atomic tokens/format strings from the DB, delete the
vocabulary-gluing logic) is gated by a **golden-master** harness. This document is the
procedure and the **allow-list of intended output changes** the gate is checked against.

## The harness

- `tests/WxReport.Tests/Golden/RendererGoldenCorpus.cs` тАФ a deterministic matrix of
  renderer inputs (38 scenarios ├Ч en/es + one unit-coverage case = **93 cases**) whose
  union exercises every renderer-reachable `LanguageTemplates` token in both languages.
- `tests/WxReport.Tests/Golden/RendererGoldenTests.cs`
  - `Verify` (`[Theory]`) тАФ renders each case and asserts byte-equality against its
    committed fixture in `Golden/Fixtures/`. Runs in CI.
  - `Record` (`[Fact]`) тАФ rewrites the fixtures; **gated** on `WX171_RECORD=1` so CI never
    rewrites the goldens it is meant to check.

### Capturing / refreshing goldens

```
WX171_RECORD=1 dotnet test tests/WxReport.Tests/WxReport.Tests.csproj \
  --filter FullyQualifiedName~RendererGoldenTests.Record
```

The fixtures committed on this branch were recorded from the **pre-rewire** renderer
(hard-coded `ReportVocabulary`). The no-regression proof: after the rewire, re-record and
`git diff` the fixtures тАФ the diff **must equal the inventory below**. Any other changed
line is an unintended regression to fix; every line below must appear (the rewire is
incomplete if an intended delta is missing).

## Coverage notes

- Every token reachable through `StructuredReportRenderer` is exercised by тЙе1 case
  (confirmed by cross-checking each token's rendered phrase against the golden union).
- **Subject-line tokens** (`ScheduledReportSubject`, `UnscheduledUpdateSubject`,
  `DiagnosticSubject`, `SubjectForConnective`, `WelcomeSubject`) are built in
  `ReportWorker`, not the renderer тАФ covered separately when that call site is rewired.
- **Currently unreachable tokens:** `sev_wx_possible`, `sev_wx_expected`. The renderer
  hard-codes "likely" for a non-convective severe day (`OutlookLikely`), so only
  `sev_wx_likely` is emitted. The other two are seeded for completeness and stay dormant
  unless the renderer is later generalized (out of WX-171 scope).

## Implementation action item (must produce NO delta)

The atomic tokens `sky_overcast_low/high`, `sky_mostlycloudy_low/high`, and `clear_and_dry`
are **lowercase** in the seed ("low overcast", "clear and dry"), but they render in cells
the current renderer shows **capitalized** ("Low overcast", "Clear and dry"). The rewired
renderer **must capitalize** the sky / weather / conditions cell (as the current code does
via the prefix word and `SkyWord`/`Capitalize`) so English output is unchanged. If the cell
is filled by raw substitution, English regresses to lowercase тАФ which would show up as an
**un-allowed** delta in the golden diff (the gate catches it). With capitalization handled,
these tokens produce no English delta; the Spanish order change above still applies.

## Post-deploy functional verification (live)

The golden gate above proves the *rendering* is correct at build time. WX-171 also changes
runtime behaviour (the DB-backed template load, the per-cycle refresh, the startup completeness
check, and the per-recipient fail-closed send gate), so it is verified **live** after deploy
(Paul's call, 2026-06-19) by `WX-171-verify.sh`.

**What it checks** (pinned to release **1.34.0**):

- **New binary is live** тАФ the service log carries the WX-171 startup line
  `Language template completeness check passed for all N loaded language(s).` and the per-cycle
  `LanguageTemplateStore loaded тАж` refresh line. Weather-independent.
- **No regression signature** тАФ *no* report body persisted since the deploy
  (`CommittedSends.EmailBody`) contains any of the OLD buggy strings the new renderer can never
  emit (`Helada lluvia`, `Bajo nublado`, `Ligera lluvia`, `lluvia probables`, en `Showers rain`,
  en `тАжwithin about 30 milesтАж`, etc.). One such hit тЗТ **FAIL** (old binary / token regression).
- **End-to-end Spanish confirmed** тАФ at least one **es** report was actually delivered, and the
  corrected phrases (`Lluvia helada`, `Nublado bajo`, `Lluvia ligera`, `lluvia probable`, тАж) are
  reported as positive evidence when the weather produced the relevant phenomena.

**Prerequisite:** the headline fix is Spanish, so PASS **requires an `es` recipient** to have
received a report since the deploy; until an es report lands the script returns **WAIT**, not
PASS. An es report arrives by one of two paths today:

- **Normal cadence** тАФ the es recipient's next **scheduled** (or an unscheduled) report. No
  restart, but you wait for the schedule.
- **On restart** тАФ the startup Diagnostic, *but only if the es recipient is the lowest-Id valid
  locality-assigned recipient* (the current Diagnostic is single-recipient; see
  `ReportWorker.SendStartupReportAsync`). **WX-199** adds a `StartupSend` flag so a Diagnostic
  fans out to every opted-in recipient (incl. es) on restart тАФ once that lands, a deterministic
  Spanish diagnostic is one restart away.

**Gate:** no time window (deterministic render). PASS needs the startup fingerprint **and** тЙе 1
es report **and** тЙе 3 report-issuing cycles (the deploy diagnostic counts as one) with zero
regression hits. The corrected Spanish is weather-dependent тАФ a dry/clear es report legitimately
carries no precip phrases тАФ so a clean run still warrants a glance at the **positive evidence**
lines to confirm a real corrected phrase was seen at least once across the cycles.

**Run** (bash/WSL, after deploy): `docs/test-procedures/WX-171-verify.sh` (override the DB with
`WX171_DB_SERVER` / `WX171_DB`; `--since` to re-anchor the boundary; `-h` for the full header).
