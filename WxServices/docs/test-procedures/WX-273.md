# WX-273 — Report-error watcher: one-time detector calibration

**This is not the watcher's "functional test" in the run-and-catch-errors sense.** The watcher is an
*instrument*: it runs continuously and reports occurrences of operator-defined error patterns in shipped
reports. The instrument's detection logic is fully covered by `ReportErrorWatcherTests`. The tickets that
*rely on* this watcher (e.g. WX-239) carry the real functional testing — see *How dependent tickets use
it* below.

What this procedure does is a **one-time calibration**: confirm that, deployed against real paths and real
data, the detector can actually produce a positive. This matters because the watcher's value is
**absence-based** — a later reader treats "no findings for pattern X over a window" as evidence the watched
error isn't occurring. That inference is only trustworthy if we've proven the detector can fire at all: a
healthy watcher reporting nothing and a silently-broken one (wrong path, permissions, SQL-dialect quirk)
look identical. So we test the smoke detector once, then trust its silence.

## Calibration (do once, post-deploy — v1.46.0)

Uses an existing real send, so it needs no waiting for a scheduled send.

1. Pick a recent real, non-diagnostic `CommittedSend` with a non-empty body and note its `SentAtUtc` and a
   distinctive phrase from its `EmailBody`.
2. Stop `WxMonitor.Svc`.
3. Edit `{InstallRoot}\Logs\wxmonitor-state.json`: set `LastReportScanUtc` to just **before** that send's
   `SentAtUtc` (so the watcher will scan it). If the key is absent, add it.
4. Create `{InstallRoot}\report-error-patterns.json` with a pattern matching the phrase from step 1, e.g.
   `[{"id":"calibration","language":"*","regex":"<phrase>","description":"one-time detector calibration","severity":"info"}]`
5. Start `WxMonitor.Svc`; wait one monitor cycle.
   *Expect:* a new line in `{InstallRoot}\Logs\findings.jsonl` with `"patternId":"calibration"`, the real
   `reportId`, `recipient`, and `language`. **This is the calibration PASS** — the detector fires against
   real data.
6. Remove `report-error-patterns.json` (or replace it with the real curated patterns). Confirm no errors in
   `wxmonitor-svc.log` and that the watcher goes inactive / picks up the new patterns cleanly.

**Test Result:** `PASS yyyy-mm-dd @ v1.46.0` once step 5 produces the line.

## How dependent tickets use it (the real functional test lives there)

A ticket that fixes or investigates a watched error class does **not** re-verify the instrument. It:

1. Adds the pattern(s) for that error's signature to `report-error-patterns.json`.
2. Lets the watcher run for a window long enough to be representative (days, given send cadence).
3. Reads `findings.jsonl`: **PASS = the watched error does not appear** (bug fixed / not occurring);
   findings = it still occurs, with the exact reports/recipients/snippets to investigate.

Because the detector was calibrated once (above), "no findings" is trustworthy silence rather than a dead
sensor.
