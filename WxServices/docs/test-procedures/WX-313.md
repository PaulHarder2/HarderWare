**What this verifies:** that WX-313 (‚â• 1.56.0) is live ‚Äî the `DbConfigurationProvider` overlays the `Config` database table onto `IConfiguration`, wired into all four services and WxManager and reloaded immediately after `EnsureSchemaAsync`. Two layers of evidence:

## 2. Probe ‚Äî a seeded key reaches configuration on the live box (the DB‚Üíconfig proof)

The substantive functional check the log alone can't show: it needs a DB write + a restart, so it's manual. It uses a throwaway key **no code consumes**, so it changes no real behaviour.

1. Seed one probe row ‚Äî **Windows PowerShell** via `sqlcmd` (`-E -C`), or SSMS, against `WeatherData`:
   ```sql
   INSERT INTO dbo.Config ([Key], [Value], [UpdatedUtc])
   VALUES (N'Diagnostics:ConfigProbe', N'wx313-probe', SYSUTCDATETIME());
   ```
2. Restart the monitor ‚Äî **Windows PowerShell (elevated):**
   ```powershell
   Restart-Service WxMonitorSvc
   ```
3. Re-run the verify script (step 1) ‚Äî or `grep` the WxMonitor log. The provider-load line over the new start should read **`loaded 1 configuration key(s)`** ‚Äî the probe row reached the provider (and therefore `IConfiguration`).
   - **Expect:** the `loaded` count is **1** (was 0 with the empty table).
4. **Clean up** ‚Äî remove the probe and restore the empty-table state:
   ```sql
   DELETE FROM dbo.Config WHERE [Key] = N'Diagnostics:ConfigProbe';
   ```
   Restart `WxMonitorSvc` again; the count returns to **0**.

- **Expect:** the `loaded` count goes `0 ‚Üí 1` while the probe is present and `1 ‚Üí 0` after cleanup ‚Äî proving the DB‚Üí`IConfiguration` path end-to-end on the live box, with no residue left behind.
