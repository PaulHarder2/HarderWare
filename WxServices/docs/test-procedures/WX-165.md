# WX-165 — Functional test procedure

**Ticket:** WX-165 — Reconciler invention → degrades (narrative dropped; diagnostic
suppressed): root-cause pass.
**Change under test:** three coordinated generation-side fixes that reduce invented
"What's changed" items at the source rather than only catching them downstream:

1. **Low sampling temperature** — the reconciliation call now sets `temperature = 0.25`
   (`ClaudeClient.InvokeReconciliationAsync`); it had run at the Anthropic default (1.0),
   which maximized variance and let each validation retry roll a fresh phantom.
2. **Diagnostic band instruction** — the `Diagnostic` report kind now gets the WX-178
   near-term-severe-onset band rule (`ForecastReconciler.BuildReconcilerSystemPrompt`); it
   had fallen through to an empty instruction, so against a stale prior it filled the band
   and the phantom degraded the (hard-aborting) startup verification.
3. **Prescriptive retry feedback** — a change-consistency rejection now replays feedback
   that **names** the offending change and says correct-or-remove, "do not replace it with
   a different change" (`ChangeConsistencyException` + the catch in `ReconcileAsync`), so a
   low-temperature retry converges instead of inventing a fresh phantom.

**Released in:** 1.31.1 · **Component:** WxReport.Svc

## Deterministic proof (the primary verification)

Unit tests in `WxServices/tests/WxReport.Tests/ForecastReconcilerTests.cs` (run with
`dotnet test WxServices/WxServices.sln`):

- `ReconcilerRequest_PinsSamplingTemperatureLow` — the request body carries
  `"temperature":0.25`.
- `DiagnosticKind_ReceivesNearTermSevereOnsetBandInstruction` — the Diagnostic kind's
  system prompt now carries the severe-onset rule ("emit an EMPTY changes array" except a
  new near-term severe hazard), no longer the empty fall-through.
- `ScheduledKind_KeepsScheduledLeadIn_NotDiagnostic` — each kind keeps its own accurate
  lead-in (a scheduled report is not told it is a diagnostic).
- `ChangeConsistencyRejection_RetryFeedback_IsPrescriptiveAndNamesChange` — on a phantom
  ("rain appearing" where prior=new rain), the retry replays the prescriptive,
  change-naming feedback ("REMOVE it from the changes array", "Do NOT replace it with a
  different change", names `'ch1'`).
- `NonChangeRejection_RetryFeedback_StaysGeneric` — a non-change rejection (a raw-UTC-block
  leak) keeps the generic "fix only that" feedback; the prescriptive text is scoped to
  change-consistency rejections.

The existing WX-148/149/151 validator tests still pass unchanged — the five change-anchored
throws are now `ChangeConsistencyException` (a `JsonException` subtype), so the
retry-then-degrade path and the exhaustion messages are identical; only the retry feedback
differs.

## Post-deploy verification (§13)

Script: `docs/test-procedures/WX-165-verify.sh` (version-pinned to 1.31.1 via
`deploy-info.sh`; 24h window). Two tiers:

### Gated — the startup diagnostic must not degrade

A deploy restarts the service, which logs exactly one startup (diagnostic) report — the
deterministic, attributable fingerprint. The script gates on:

- **PRECONDITION / EXERCISED:** a `sending startup (diagnostic) report` line in the window.
- **REGRESSION (fail) signature:** `reconciliation failed for startup send: degraded` —
  the very abort WX-165 prevents for the phantom-band case. Expect **0**.
- Confirmation: a `startup (diagnostic) report sent` line.

```bash
bash docs/test-procedures/WX-165-verify.sh
```

*Expect:* once a startup diagnostic has run and 24h of cycles have accrued with no startup
degrade → **PASS**. A startup degrade → **FAIL** at any horizon. No startup diagnostic in
the window yet → **WAIT**.

### Reported — the general degrade rate (human judgment)

The temperature + retry-feedback fixes should also cut the *general* reconciler degrade
rate. "Materially fewer" is a statistical read over a noisy background, so the script
**reports** (does not gate on) the pre/post degrade counts and the per-day rate:

- *Expect:* the post-deploy reconciler degrade rate is **materially below** the pre-fix
  baseline of **~16.5/day** (the ~33 events / 2 days observed before WX-165). A flat or
  higher rate means the generation-side fixes are not converging — investigate even when
  the startup-degrade gate passes.

Cross-check the §13 baseline smoke (services up, heartbeat current, 1.31.1 running, no new
ERRORs above the equal-length pre-deploy window) — the verify script's BACKGROUND HEALTH
section reports the ERROR delta.
