# WX-329 Test Procedure — retire the installer, the release-packaging script, and the native-service fallback

**What this verifies:** that the four services still build, start, run and stop correctly now that
the Windows-service host integration has been removed from them; that no packaging machinery remains
for a distribution that does not exist; and that the documents describe the install root a deployment
actually produces rather than one an installer used to unpack.

**Scope grew during the ticket.** It began as "retire the Inno Setup installer" and became the whole
native-service retirement, because each piece turned out to hold up the next: the installer's only
purpose was registering native services; `Build-Release.ps1` existed only to feed the installer; and
the services still carried `UseWindowsService` for a fallback we have now decided against. The one
genuinely risky change is the last — it alters the shipped binaries — so steps 4-5 are the heart of
this procedure and the earlier steps are cheap confirmations.

**Run it interactively** — read one step, perform it, report the result; note failures and continue.
The list of failed steps at the end is the run's deliverable (WORKFLOW §7a point 4).

**Recording results:** **Test Proc** = `docs/test-procedures/WX-329.md`; **Test Result** =
`PASS yyyy-mm-dd @ <version>` (or `FAIL yyyy-mm-dd — <what>`).

## 1. The retired machinery is gone

From the repo root:

```shell
git ls-files | grep -iE "HarderWare_WxServices\.iss|PRE-INSTALL\.txt|Build-Release\.ps1"
```

- *Expect:* **no output** — none of the three is tracked any more.

```shell
git grep -nIE "Inno Setup|HarderWare_WxServices|\bISCC\b|PRE-INSTALL\.txt|DAppVer|Setup\.exe|Build-Release" -- ':(top)' ':(top,exclude)WxServices/VERSIONS.md' ':(top,exclude)WxServices/docs/test-procedures/*'
```

Read the pattern before running it — four details are load-bearing, each wrong in an earlier draft:

- **`:(top)`, not a bare `.`** — plain `':!WxServices/VERSIONS.md'` exclusions are *cwd-relative*, and
  git **silently ignores** a pathspec that matches nothing. Run from `WxServices\` with the earlier
  form, the exclusions evaporated and unrelated files flooded in — a failure reading, with nothing
  wrong.
- **`PRE-INSTALL\.txt`, not bare `PRE-INSTALL`** — the bare form matches the ordinary word
  "pre-**install**ed".
- **`Inno Setup`, not bare `inno`** — the bare form matches "**inno**cent", which appears in two
  unrelated files. `-I` skips binaries; it is belt-and-braces, not load-bearing (nothing binary
  matches the pattern as written).
- **`docs/test-procedures/*` excluded wholesale** — this file quotes every term it searches for, and
  older procedures (`WX-65.md`) record the native-service era as history. Historical records are not
  rewritten.

- *Expect:* a small number of hits, all of them **descriptive history** — the *Distribution: there is
  none* section of `DESIGN.md`, which names both retired scripts so a reader who meets them in history
  knows what became of them, and `Setup.exe` in `INSTALL.md`'s banner and Uninstall section, which
  exist for a reader who still has an installation made by the old installer. **The count is
  deliberately not pinned** — legitimate prose edits move it, and a runner taught to match a number
  will edit the number instead of reading the hit.
- *Expect (the real invariant):* **no hit is an instruction to build, run, or ship anything** — no
  `ISCC` command line, no `.iss` compile step, no `/DAppVer`, no `.\Build-Release.ps1` invocation.

## 2. No built installer survives on disk

Step 1 greps for *references*; this checks for the *artifact*. They are different questions, and on
the development box the answer differed: a 44 MB `HarderWare_WxServices_Setup.exe` built 2026-04-11
was sitting in `WxServices\installer_output\`, hidden by an `installer_output/` line in `.gitignore`
that outlived the script it belonged to. Removing that line is what exposed it — a reference sweep
would never have found it, because a gitignore entry is not a reference, it is a rule for *not*
seeing things.

```shell
# (a) the ignore rule is gone - the unambiguous check
grep -rn "installer_output" .gitignore WxServices/.gitignore .dockerignore

# (b) check-ignore, with a control proving the check can still detect a live rule
git check-ignore -v WxServices/installer_output/x.exe 2>&1 | grep -q . && echo "STILL IGNORED - bad" || echo "no ignore rule"
git check-ignore -v WxServices/test_reports.txt      2>&1 | grep -q . && echo "control OK - check can detect a live rule" || echo "control FAILED - check detects nothing"

# (c) the ARTEFACT itself - these are the commands the Expects below actually rest on
find . /mnt/c/HarderWare -iname "*WxServices_Setup*" -o -type d -name "installer_output" 2>/dev/null
test -d WxServices/release && echo "release/ EXISTS - bad" || echo "no release/ directory"
```

- *Expect:* the `grep` finds **nothing** — that is the unambiguous check, because `check-ignore`
  alone is not trustworthy here: the old rule was `installer_output/`, and a **trailing-slash pattern
  only matches directories**, so once the directory is gone `check-ignore` reports "not ignored"
  whether or not the rule still exists. Naming a *file* inside the directory (`.../x.exe`) sidesteps
  that; the `grep` settles it outright.
- *Expect:* `no ignore rule`, then `control OK`. The control is the negative control: without it, a
  `check-ignore` that silently matched nothing would read exactly like a clean result. Use
  `test_reports.txt` — a plain file pattern — precisely because a directory pattern would reintroduce
  the trap above. (The first control this procedure tried was `WxServices/obj`, which reported
  "not ignored" despite `obj/` being in `.gitignore`, for exactly that reason.)
- *Expect:* block (c) prints **nothing** from `find`, then `no release/ directory`. This is the check
  that actually answers the question — an earlier draft asserted "no installer survives on disk" while
  running only ignore-rule and git-tracking commands, so a stale **untracked** directory or a
  leftover `Setup.exe` would have passed unnoticed. The assertion now has a command behind it. The one that existed was deleted under
  this ticket (agreed with Paul 2026-07-21): it installs the pre-container system, registering four
  services that no longer exist, so running it could only break a machine.

**The two ignore files were deliberately left asymmetric**, so do not "fix" one to match the other:

| File | `release/` entry | Why |
|---|---|---|
| `WxServices/.gitignore` | **removed** | Nothing produces the directory now. Leaving it ignored is what let a stale tracked copy hide; with no rule, any reappearance shows up in `git status` where a human sees it. |

Cheap insurance against a real cost. The bare `release/` this replaced lacked the `**/` prefix the file's own header mandates, so it never matched `WxServices/release/` — it was broken insurance, not insurance. |

## 4. The services build without the Windows-service host integration

This is the change that alters the shipped binaries.

**Read the branch's CI run rather than rebuilding locally.** `.github/workflows/ci.yml` runs the
identical `dotnet build`/`dotnet test` on `WxServices.CI.slnf` for every push, *plus*
`dotnet format --verify-no-changes` and a `linux-x64` publish of all four services — which is the
more relevant evidence for a change to containerized binaries than a Windows Release build
(see the local-green-is-not-CI-green trap). Run the commands locally only if the branch carries
changes beyond those enumerated here.

- *Expect:* the PR's CI job is green, including the format check and the four Linux publishes.
- *Expect:* this returns **nothing** — the host call, the `Microsoft.Extensions.Hosting.WindowsServices`
  package reference, and the vestigial `<WindowsService>` MSBuild property are gone from all four
  services:

```shell
  git grep -nE "UseWindowsService|WindowsServices|<WindowsService>|sc\.exe" -- ':(top)WxServices/src'
```

  **The pathspec is required.** Without it the same command run from the repo root returns hits in
  `DESIGN.md`, `Deploy-WxService.ps1`, `VERSIONS.md` and this file — all legitimate prose — and a
  runner comparing that to an Expect of "nothing" records a false FAIL. An earlier draft of this step
  omitted it, which is the third time the same pathspec mistake appeared in this one procedure.
- *Expect:* the `sc\.exe` alternative above returns nothing under `src/` either. It is there because
  the mechanism and the *instructions for using it* live in different places: this ticket removed
  `UseWindowsService` but initially left `Install-Service.ps1` (a working `sc.exe create` script) and
  an `sc.exe` install recipe in the header comment of all four `Program.cs`. Grepping only for the
  host call would have passed while the tree still told a reader to register the service.

**Why the explicit `Microsoft.Extensions.Hosting` reference appeared.** Removing the Windows-services
package broke the build with `The name 'Host' does not exist in the current context` — that package
had been supplying `Microsoft.Extensions.Hosting` transitively, and every service's
`Host.CreateDefaultBuilder` depended on it without saying so. Each service now references the generic
host directly. If this step fails to compile, that dependency is the first place to look.

## 5. The containers still build, start, run and stop

The binaries changed, so this is not a formality.

### 5a. Pre-flight — host construction, without touching production

Run this **before** disturbing anything. It answers the one question the change actually risks — does
the generic host still build without `UseWindowsService`? — and it cannot affect the running system.

From **Windows PowerShell** (the volume path is a Windows path; the same command from WSL fails with
`invalid volume specification`):

```shell
docker compose build     # from services\ ; builds images, does NOT recreate running containers
docker run --rm --network none `
  -v 'C:/Code/HarderWare/services/wxparser/appsettings.local.json:/opt/wxservices/appsettings.local.json:ro' `
  wxparser:dev
```

**Why this shape, and why not `docker compose run`.** Review suggested driving the pre-flight through
compose so it inherits the real mounts and environment — the concern being that a bare `docker run`
lacks the config, so a failure *before* `Migrator.MigrateAsync` could be a missing-setup artefact
rather than a broken host. The concern is right; the remedy is not. `docker compose run wxparser`
starts a **second fully-wired WxParser** against the live database and shared `Logs\`/`plots\` — a
dual writer beside the running container, the exact collision `Deploy-WxService.ps1`'s cutover block
exists to prevent. It was tried while writing this step: the one-off container came up *healthy* and
had to be killed. It also never exits, because a healthy worker service runs forever.

So: keep the isolation (`--network none`, no shared mounts, `--rm`) and answer the ambiguity by
mounting the **real** `appsettings.local.json` read-only. Real configuration, zero blast radius —
the minimum-privilege shape used for any ad-hoc verification against this system.

- *Expect:* all four images build.
- *Expect:* the container gets **past host construction** and reaches `Migrator.MigrateAsync`, then
  fails on the database with a `SqlException … error: 26`. Reaching EF migration proves
  `Host.CreateDefaultBuilder` succeeded, configuration loaded, and startup ran — everything
  `UseWindowsService` used to sit next to, which is the whole question this step exists to answer.
  The SQL failure is the **expected** consequence of `--network none`, not a defect.
- *Failure signature to watch for:* an exception **before** any DB activity — a missing-method or
  missing-assembly error naming `Microsoft.Extensions.Hosting` — which would mean the explicit
  generic-host package reference did not take.
- *Expect:* `docker ps` still shows the four production containers untouched, and
  `docker ps -a --filter name=-run-` is empty (no stray one-off container left behind).

### 5b. The live stack, via the real deploy

Do this as **one** cycle, not two: `Deploy-WxService.ps1` already runs `docker compose up -d --build`
and verifies each container started, so a separate `compose up --build` here would build and start
everything twice for the same evidence. From the `WxServices` directory, in an **elevated**
PowerShell:

```powershell
.\Deploy-WxService.ps1 all
```

- *Expect:* it completes; all four containers redeploy and verify started; WxManager and WxViewer
  publish into `{InstallRoot}`; `{InstallRoot}\Logs\deploy-history.log` records `OK` for each.
- *Expect:* **no** "Disabling leftover native … service" line — there are no native services left to
  disarm. That block is retained only for a machine predating the cutover (the ~2026-08-17 PC
  migration); on a current box it must be a silent no-op.
- *Expect:* all four stay running — re-check `docker compose ps` after two minutes. A container that
  starts then exits is the signature of a broken host lifetime.
- *Expect:* each service's log in `{InstallRoot}\Logs\` shows its startup banner and normal
  operation, with no startup exception.
- *Expect:* within one fetch cycle (allow ~10 minutes), fresh METAR data appears — proof the services
  are doing real work, not merely staying up.

### 5c. Graceful shutdown

The one property `UseWindowsService` would have configured, and so the one most worth checking.
From `services\`:

```powershell
docker compose stop
docker compose logs --tail 30
docker compose start
```

- *Expect:* the containers stop **without** being force-killed at the 10-second timeout, and the logs
  show orderly shutdown rather than an abrupt end; then all four return to running.
- *Why verify what should be a no-op:* `UseWindowsService()` early-returns when
  `WindowsServiceHelpers.IsWindowsService()` is false, so in a Linux container it never touched the
  lifetime, the content root, or logging — the console lifetime has always handled SIGTERM. That is
  the assumption the entire change rests on, which is exactly why it is demonstrated rather than
  asserted.

## 6. The documents describe the install root that deployment actually produces

`INSTALL.md` §3 was rewritten because its tree was inherited from the installer era and no longer
matched reality — it listed `services\`, `observability\`, and a top-level `appsettings.shared.json`,
none of which are there.

Compare `INSTALL.md` §3's tree against the real install root (`Get-ChildItem C:\HarderWare`, or your
`InstallRoot`):

- *Expect:* every directory §3's tree lists is present. **The converse does not hold on this box**,
  and §3 says so: a machine that predates the containerization also carries `tools\`, `WxVis\`,
  `services\`, `wgrib2\` or `observability\`, which §3 names as inert installer-era leftovers. Verify
  the *claim* rather than the directory listing — confirm nothing creates or reads them: no
  `New-Item`/publish target in `Deploy-WxService.ps1` produces them, and the only `WxPaths` members
  pointing at `WxVis\`/`services\` are consumed by containerized code (resolving to in-container
  paths) or by the setup console against the **source** tree.
- Ignore install-root entries belonging to other projects and to operational state (`backup*`,
  `BuildCache`, `setup`, `autoheal`) — §3 documents what a WxServices deploy creates, not everything
  that has ever accumulated in the directory.
- *Expect:* the entries §3 attributes to the deploy really are produced by it — `Logs\` explicitly
  (`Deploy-WxService.ps1`), `WxManager\`/`WxViewer\` by `dotnet publish -o`, and `plots\` /
  `translation-qa\` by Docker creating the bind-mount sources. `temp\` is described separately and
  correctly as appearing only when you **save** on WxManager's Configure tab
  (`ConfigureTab.xaml.cs`) — not at deploy, not at launch. This distinction is the whole point: the
  tree §3 replaced was inherited from the installer and listed directories a deploy never creates.
- *Expect:* `appsettings.shared.json` and `log4net.shared.config` are **not** at the top level but
  **are** beside each application's binaries (`WxManager\appsettings.shared.json`), exactly as §3 now
  states.
- *Expect:* §3 describes deployment creating the install root — no "extract the distribution
  archive", because there is no archive.

`DESIGN.md` → *Distribution: there is none (WX-329)*:

- *Expect:* it states there is no installer and no distributable package, records what was retired and
  why, and describes the source-checkout → setup-console → `Deploy-WxService.ps1` path.
- *Expect:* it no longer claims the installer "registers no Windows services" — that was false (the

- *Expect:* the *Prerequisites* list says containers are the only runtime, and marks host

  §5. Before this ticket it told the reader to start `WxParserSvc` first and wait a cycle — advice for

- *Expect:* the legacy `Setup.exe` step comes **before** "delete the installation directory". The

- Check this machine: **Windows Settings → Apps → Installed apps**, search *HarderWare*. Either answer
  passes, and what you find selects which claim you verified:
  - **No entry** — the common case, and why the wording had to become conditional.
  - **An entry exists** — follow the conditional branch and confirm it is executable. *Do not fail the
