# WX-160 Functional Test Procedure — TAF-aware significance gate (post-deployment verification)

**What this verifies:** that WX-160 (the TAF-aware significance gate, ≥ 1.26.0)
is live and behaving in production — the `taf-fresh` bypass is retired, the gate
suppresses unchanged TAF cycles deterministically (instead of calling Claude),
genuine changes still reach Claude, the windKt sustained-only guard holds, and no
new errors were introduced. It is the deployment-side check the 575
unit/integration tests cannot give (real GFS/TAF/METAR inputs, real cadence), and
it doubles as the standing regression check for the gate after any later redeploy.

**Recording results:** when the run passes, set the ticket's custom fields —
**Test Proc** = `docs/test-procedures/WX-160.md`, **Test Result** =
`PASS yyyy-mm-dd @ <version>`. On failure: `FAIL yyyy-mm-dd @ <version> — <which
check>`, a ticket comment describing it, and open the Bug (the
`FAIL: Open bug ticket` transition).

---

## 0. Run the verification

The check is a **bash** script (not PowerShell) — `WX-160-verify.sh`, beside this
procedure — that reads the service log and `deploy-history.log`. The only
requirement is *bash* with read access to those logs; on the HarderWare PC that
means **WSL**, which supplies bash and the `/mnt/c` view the default paths use
(any other bash works if you point `--log`/`--deploy-log` at the logs). Run it
from the repo root in WSL bash:

```bash
bash WxServices/docs/test-procedures/WX-160-verify.sh
```

From a **PowerShell** prompt, shell out to WSL (also from the repo root — `wsl`
inherits the current directory, so the relative path resolves):

```powershell
wsl bash WxServices/docs/test-procedures/WX-160-verify.sh
```

No arguments needed: the script auto-detects the deploy boundary from the **latest
`WxReportSvc` line in `deploy-history.log`** and reports the deployed
version + commit. `--since 'YYYY-MM-DD HH:MM:SS'` (UTC) overrides the boundary;
`--log` / `--deploy-log` point at non-default paths.

**When to run:** after any deploy that includes WX-160 (≥ 1.26.0), as the
**In Test** gate before `PASS: Send to Done`. **Give it a full active day**
post-deploy before the PASS/FAIL call — the signal accrues over live cycles; a
quiet hour shows little.

**Expect:** the report prints, headed by
`Deployed: WxReportSvc <version> (commit <hash>)` matching the release under test.

Throughout: "**Expect:**" lines are the pass/fail criteria. If reality differs,
stop, note which check, and record a FAIL.

---

## 1. New code is live

In the report's **NEW CODE LIVE?** block:

1. **Expect:** `"TAF-merged" suppression wording present` > 0 **and**
   `"fired: taf-fresh" lines after deploy` = **0**.
2. **Expect:** the `Deployed:` line shows the expected version and commit.

A non-zero `taf-fresh` count, or a wrong/old version, means the previous binary
is still running → **FAIL** (re-check the deploy/restart).

## 2. The gate is making real decisions

In **GATE BEHAVIOR SINCE DEPLOY** and the `gate-passed 'fired:' reasons` list:

1. **Expect:** `Gate SUPPRESSED` > 0 over the window — the deterministic
   TAF-merge suppression replacing what used to be `taf-fresh` Claude calls.
2. **Expect:** every gate-passed `fired:` reason is a **real** criterion
   (precip / wind / severe / temperature) — **never** `taf-fresh`.

## 3. taf-fresh collapse

In **TAF-FRESH COLLAPSE** (equal-length before/after windows):

1. **Expect:** after-deploy `taf-fresh` = **0**, against a non-zero before-deploy
   count — the bypass is retired.

## 4. Health

In **WINDKT VALIDATOR** and **HEALTH**:

1. **Expect:** `ERROR lines` = 0 and `Reconciliation degraded/inconsistent` = 0.
2. `Folded-gust rejections` > 0 is **fine** — that is the windKt guard catching
   Claude fold a gust into `windKt.max` and correcting it on retry. Only a flood,
   or any that ends in an ERROR/degrade, is a concern.

## 5. Verdict & decision

The `VERDICT` block always ends with exactly one explicit token —
**`PASS`**, **`FAIL`**, or **`WAIT`**. The token is the *mechanical* gate the
script can compute deterministically; on a `PASS` you still owe the one human
judgment the script cannot make (step 4 below).

1. **`PASS`** (token): `taf-fresh` retired (0), the gate is making real decisions
   (suppressed/passed > 0), and health is clean (no ERROR / reconciliation-
   degraded lines). This is the *necessary* condition — not yet the whole decision.
2. **`FAIL`** (token → `FAIL: Open bug ticket`): `taf-fresh` still firing (old
   binary), **or** any new ERROR / reconciliation-degraded line since deploy. The
   script is fail-closed on either. File a Bug linked to WX-160; this ticket still
   completes (Done → Closed), per the fix-forward rule (WORKFLOW.md §13).
3. **`WAIT`** (token): not enough signal yet — no TAF/GFS-driven cycle has reached
   the gate, so there is nothing to judge. **Give it a full active day** and
   re-run; do not pass or fail on a `WAIT`.
4. **Human confirmation before `PASS: Send to Done`** (the token can't compute
   this): read §2's `gate-passed 'fired:' reasons` — every one must be a real
   criterion (precip / wind / severe / temperature), never `taf-fresh` — and the
   sample suppressions, which should be genuine no-change cycles. A `PASS` token
   with a suspicious reason or a wrongly-suppressed real change is a **FAIL** by
   judgment: open the Bug. Only a `PASS` token *and* this confirmation send to Done.
