# WX-256 — noon/midnight soft tokens: post-deploy verification

**Change (v1.43.0, WxReportSvc).** `noon`/`midnight` are curated `LanguageTemplates` tokens (`CultureInfo` has no such designator, so the truth lives in the DB — en-seeded, top-up-generated + QA-reviewed for targets). But a cosmetic time word must not fail-closed like a hazard: a new **`Tok.Soft`** allowlist `{ noon, midnight }` is subtracted from the suppression set — **`Tok.Required` = `Tok.All` − `Tok.Soft`** — and the two send gates + the startup completeness check key on `Tok.Required`. So a language missing **only** soft tokens still **sends** (the renderer degrades via `TemplateSet.Has` to the culture 12-hour form) while WX-250 top-up fills the words; hazard tokens stay hard fail-closed.

**Boundary.** The 1.43.0 `WxReportSvc` deploy — `deploy-info.sh --version 1.43.0 WxReportSvc`.

**Gate (agreed with Paul, 2026-07-04).** The **mechanism-proven** bar: past a 10-min window, cycles ran, no language was suppressed over the soft tokens, and top-up filled at least one target's words. Full five-language fill (~25 min) just repeats it and is noted, not gated on.

---

## 1. Automated check

After the 1.43.0 deploy, run:

```
bash docs/test-procedures/WX-256-verify.sh
```

It resolves the deploy boundary (version-pinned) and, over the `WxReport.Svc` log since that boundary, reports:

- **Soft-token suppression** — any `ERROR` naming `noon`/`midnight` (an INCOMPLETE / per-recipient suppression). *Expect:* **none** — that is the whole payoff.
- **Top-up filling the words** — `WX-250: '<iso>' topped up …` lines; post-1.43.0 the only missing baseline tokens are noon/midnight, so a target top-up **is** the fill. *Expect:* ≥1 within ~1–2 cycles.
- **Exercised** — `Report cycle complete.` lines (the gate/send path ran).
- **Background health** — new `ERROR` lines vs the equal-length pre-deploy window.

**Verdict:** `PASS` once past the 10-min window with cycles run, ≥1 target filled, and no suppression; `WAIT` before the deploy or before the first fill; `FAIL` if a soft token suppressed a report. Record the printed `Test Result` string in the Jira **Test Result** field.

## 2. Manual checks (the deep ones the log can't express)

**2a. The words actually landed for the targets (DB).** In **PowerShell**:

```
sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -Q "SET NOCOUNT ON; SELECT l.IsoCode, MAX(CASE WHEN t.Token='noon' THEN 1 ELSE 0 END) AS [noon], MAX(CASE WHEN t.Token='midnight' THEN 1 ELSE 0 END) AS [midnight] FROM Languages l LEFT JOIN LanguageTemplates t ON t.LanguageId=l.Id AND t.Token IN ('noon','midnight') AND t.Representable=1 WHERE l.IsEnabled=1 GROUP BY l.IsoCode ORDER BY l.IsoCode;"
```

*Expect:* `en` = `1/1` immediately (seeded by the migration); each enabled target reaches `1/1` within ~25 min as top-up fills it (one language per ~5-min cycle).

**2b. A rendered noon reads the word.** Inspect a report or welcome whose local time hits the boundaries (add a test recipient and view the welcome is the easiest trigger).

- *Expect (event/body):* an `en` report whose obs or a `{q:time}` lands at local noon shows "…, noon" — **not** "12:00 PM". Midnight in the body stays the date-bound "12:00 AM" (no midnight word).
- *Expect (schedule/welcome):* the welcome reads "… and **12:00 noon**"; a midnight send hour reads "**12:00 midnight**"; a non-noon hour reads the precise "6:00 AM" form.
- *Expect (target):* once filled, a target-language report renders **its** noon word (es "mediodía", …).

**2c. No hazard regression (safety not weakened).** Confirm a normal severe report still renders and its hazard words appear — hazard tokens stayed hard-required (`Tok.Required`), so a language missing a hazard token still suppresses.
