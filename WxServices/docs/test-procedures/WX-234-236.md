# WX-234 + WX-236 — Extended Forecast grid: chronological Low→High order, and no partial-day pseudo-extreme

**Type:** Functional verification (post-deploy), **manual spot-check** of the diagnostic report's Extended Forecast grid. Both fixes are deterministic and unit-test-proven (`StructuredReportRendererTests`: the WX-236 ordering test and the WX-234 partial-day em-dash tests; the shared-gating test pins grid and summary to one `DayPartBands` rule); this confirms them live in a real rendered report. The grid HTML is also persisted in `CommittedSends.EmailBody`, so the optional deeper step can grep the fingerprint rather than eyeball it.

**Release:** 1.40.1
**Components:** WxReportSvc — `StructuredReportRenderer.AggregateDays` (extreme gating + nullable High/Low), the grid temperature cell (Low-before-High order, em-dash for a suppressed extreme), and shared `DayPartBands` gating.

## Background

Two editorial-clarity fixes to the same grid cell, shipped together (one branch, one golden re-record):

- **WX-236 (order):** the grid's daypart columns run chronologically left-to-right (00-06 … 18-24 local), but the temperature pair was printed **High then Low**. The Low we report is this morning's minimum — the *earliest* point of the day — so High-then-Low on a chronological strip read as "today's high, then *tomorrow-morning's* low." The pair is now **Low then High**, matching the time-arrow.
- **WX-234 (partial-day extreme):** every report's GFS horizon ends on a partial trailing local day with only its pre-dawn/morning block (no afternoon/peak-heating block); the grid showed that day's morning max as the daily **High** (e.g. a steady 98-99 °F week ending in a bogus `High: 90°F`). The grid now gates each extreme on the band that holds it — a **High** needs an afternoon (local 12:00-18:00) block, a **Low** needs a pre-dawn/morning (local 00:00/06:00) block — via the same `DayPartBands` predicate the WX-230 temperature summary uses, so grid and prose cannot drift. A suppressed extreme renders as an **em dash** (`—`); the row keeps its other extreme, wind, and conditions.

## Preconditions

- 1.40.1 deployed (`deploy-history.log` has a `WxReportSvc 1.40.1` line).
- A delivered report (the routine diagnostic report is ideal) whose forecast horizon ends on a partial (pre-dawn/morning-only) last local day — i.e. essentially any normal report.

## What a given email does and doesn't exercise

The two fixes are exercised differently by a single Diagnostic report:

- **WX-236 (order) is exercised by EVERY report** — every grid row carries a Low/High pair, so the Low-above-High order is always verifiable.
- **WX-234 (em dash) is exercised only when the grid actually contains the relevant partial day**, and *which* partial days appear depends on the **send time** (where "now" sits within the local day) and the **GFS horizon** — not on the code. So one Diagnostic email may contain a trailing partial day, a leading partial day, both, or **neither**:
  - **Absence-of-max** is testable only if the grid's **last** day is a trailing partial (its horizon ends before that day's afternoon block).
  - **Absence-of-min** is testable only if the grid's **first** day is a leading partial (today, whose snapshot has no pre-dawn/morning block — the rarer case, since a retained day usually keeps its elapsed morning block per WX-176).

Read the **Conditions cell** to tell whether a row is partial (see step 3). If the case you want isn't present in this email, that is expected — see *If a case isn't exercised this run*.

## Procedure

1. **Open the post-deploy diagnostic report email and find the "Forecast for …" Extended Forecast grid.**

2. **WX-236 — order (always exercised, every row).** In each day's Temperatures cell, confirm the **Low** line is above the **High** line.
   - **Expect:** every cell reads `Low: <t>` first, `High: <t>` second. No cell is High-first. *(If this fails, the test FAILS regardless of the partial-day cases below.)*

3. **Identify which partial days this email actually contains** — the Conditions cell names each row's present bands:
   - **Trailing partial (tests absence-of-max):** the **last** row's Conditions cell shows only early bands (e.g. `00-06 …, 06-12 …`) with **no** `12-18`/`18-24` band → do step 4a. If the last row carries a `12-18` or `18-24` band, this email does **not** exercise absence-of-max.
   - **Leading partial (tests absence-of-min):** the **first** row is today and its Conditions cell shows only `12-18`/`18-24` with **no** `00-06`/`06-12` band → do step 4b. If the first row carries a `00-06`/`06-12` band, this email does **not** exercise absence-of-min.

4a. **Absence-of-max — the trailing partial row.**
   - **Expect:** that row reads `High: —` (em dash); its **Low**, wind, and conditions remain numeric/present.

4b. **Absence-of-min — the leading partial row.**
   - **Expect:** that row reads `Low: —` (em dash); its **High**, wind, and conditions remain numeric/present.

5. **Full days unaffected (always check).** Every row whose Conditions cell carries an afternoon (`12-18`) band shows a numeric **High**, and every row carrying a `00-06`/`06-12` band shows a numeric **Low**.
   - **Expect:** no full day shows an em dash for either extreme; only a partial day suppresses the extreme it cannot support.

6. **(Optional, scriptable) Grep the persisted body instead of eyeballing.** Pull the latest diagnostic send's grid HTML (PowerShell, local SQL):
   ```powershell
   sqlcmd -E -C -S .\SQLEXPRESS -d WeatherData -Q "SET NOCOUNT ON; SELECT TOP 1 EmailBody FROM CommittedSends WHERE IsDiagnostic = 1 ORDER BY Id DESC" -y 0
   ```
   - **Expect:** every grid temp cell matches `Low: …<br/>High: …` (Low before High) and **none** matches `High: …<br/>Low:` (old order gone — unconditional). A `High: —` / `Low: —` match appears only if the email contained the matching partial day (conditional, per step 3).

## If a case isn't exercised this run

A single Diagnostic email may not contain the partial day needed to exercise absence-of-max and/or absence-of-min — **that is expected**, not a failure, because which partial days appear is a function of send time and the GFS horizon, not the code. The deterministic unit tests (`Grid_PartialDay_SuppressesUnsupportedExtreme_AsEmDash`, `Grid_And_Summary_GateExtremesByTheSameRule`, `DayPartBands_GatesAfternoonAndDawnWindow`) prove both branches; this live check confirms whichever partial day the email happens to contain.

To force a live confirmation of a case that didn't appear, **restart the WxReportSvc service** so it builds a fresh Diagnostic report at a shifted "now" (and possibly a newer GFS horizon), then re-inspect the new email. Repeat until the case appears (the trailing-partial *absence-of-max* case is the common one; *absence-of-min* may need a send time when today's snapshot starts at its afternoon block).

## Result

Record in **each** ticket's Jira **Test Result** field (WX-234 and WX-236): `PASS yyyy-mm-dd @ 1.40.1`, **noting which cases were exercised live** vs. left to the unit tests — e.g. `PASS 2026-mm-dd @ 1.40.1 — order + trailing absence-of-max confirmed; leading absence-of-min not present this run, covered by unit test`.
