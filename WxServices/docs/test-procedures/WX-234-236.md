## Background

- **WX-236 (order):** the grid's daypart columns run chronologically left-to-right (00-06 â€¦ 18-24 local), but the temperature pair was printed **High then Low**. The pair is now **Low then High**, matching the time-arrow.

## What a given email does and doesn't exercise

The two fixes are exercised differently by a single Diagnostic report:

- **WX-236 (order) is exercised by EVERY report** â€” every grid row carries a Low/High pair, so the Low-above-High order is always verifiable.
- **WX-234 (em dash) is exercised only when the grid actually contains the relevant partial day**, and *which* partial days appear depends on the **send time** (where "now" sits within the local day) and the **GFS horizon** â€” not on the code. So one Diagnostic email may contain a trailing partial day, a leading partial day, both, or **neither**:
  - **Absence-of-max** is testable only if the grid's **last** day is a trailing partial (its horizon ends before that day's afternoon block).
  - **Absence-of-min** is testable only if the grid's **first** day is a leading partial (today, whose snapshot has no pre-dawn/morning block â€” the rarer case, since a retained day usually keeps its elapsed morning block per WX-176).

Read the **Conditions cell** to tell whether a row is partial (see step 3). If the case you want isn't present in this email, that is expected â€” see *If a case isn't exercised this run*.

## If a case isn't exercised this run

A single Diagnostic email may not contain the partial day needed to exercise absence-of-max and/or absence-of-min â€” **that is expected**, not a failure, because which partial days appear is a function of send time and the GFS horizon, not the code. The deterministic unit tests (`Grid_PartialDay_SuppressesUnsupportedExtreme_AsEmDash`, `Grid_And_Summary_GateExtremesByTheSameRule`, `DayPartBands_GatesAfternoonAndDawnWindow`) prove both branches; this live check confirms whichever partial day the email happens to contain.

To force a live confirmation of a case that didn't appear, **restart the WxReportSvc service** so it builds a fresh Diagnostic report at a shifted "now" (and possibly a newer GFS horizon), then re-inspect the new email. Repeat until the case appears (the trailing-partial *absence-of-max* case is the common one; *absence-of-min* may need a send time when today's snapshot starts at its afternoon block).
