# WX-314 Test Procedure — first-time setup script (`WxServices.Setup`)

**Authored test-first (2026-07-18):** this spec was written *before* the code, as the check on what we build (WX-314's ACs 1–7 made concrete). The unit tests (§0) are written next and drive the generators; the code is then written to make them pass; the functional isolated-run (§1) is executed after the code exists.

**What this verifies:** that `WxServices.Setup` configures a fresh box end-to-end — validates prerequisites, provisions the least-privilege SQL login, writes the five per-environment `appsettings.local.json` files, ensures the schema, and seeds the foundational fields into the `Config` table — **fully parameterized** so a test run touches only isolated, throwaway targets and leaves the real instance/root untouched.

**Recording results:** **Test Proc** = `docs/test-procedures/WX-314.md`; **Test Result** = `PASS yyyy-mm-dd` (setup tooling — no runtime version). On failure: `FAIL yyyy-mm-dd — <what>`, a ticket comment, and a Bug.

---

## 0. Unit tests (CI — the pure generators; no DB, no filesystem)

`tests/WxServices.Setup.Tests`. These pin the deterministic input→output surface (AC-5, AC-6):

1. **Container connection string** — `BuildContainerConnectionString(db, login, password)` →
   `Server=host.docker.internal,1433;Database=<db>;User Id=<login>;Password=<password>;TrustServerCertificate=True;`
   - *Expect:* exact string; `<db>`/`<login>`/`<password>` substituted; server is always `host.docker.internal,1433` (the container→host route, not `--server`).
2. **WxManager connection string** — `BuildWxManagerConnectionString(server, db)` →
   `Server=<server>;Database=<db>;Trusted_Connection=True;TrustServerCertificate=True;`
   - *Expect:* exact string; native Trusted auth, **no password**.
3. **Container `local.json` from template** — `BuildContainerLocalJson(exampleJson, connectionString)` loads the committed `.example`, replaces `ConnectionStrings:WeatherData` with the built string, and returns it.
   - *Expect:* `ConnectionStrings:WeatherData` == the built string (placeholder gone); **`_README` blocks, `Telemetry`, and per-service extras (`Gfs:Wgrib2Path` / `WxVis:CondaPythonExe` / `Monitor:Grafana*`) preserved unchanged**; valid JSON.
4. **WxManager `local.json`** — `BuildWxManagerLocalJson(connectionString)` →
   - *Expect:* `{ "ConnectionStrings": { "WeatherData": "<trusted string>" } }`; **no** `Fetch:*`/`WxVis:MapExtent` keys (those are now DB-seeded, not file — AC-6).
5. **Foundational `Config` seed rows** — `BuildFoundationalSeedRows(inputs)` →
   - *Expect:* exactly the rows `Fetch:HomeIcao`, `Fetch:HomeLatitude`, `Fetch:HomeLongitude`, `Fetch:BoundingBoxDegrees`, `Fetch:RegionSouth/North/West/East`, `WxVis:MapExtent` — keyed in the `Section:SubKey` convention, values from the inputs. **No** bootstrap-critical keys present (`ConnectionStrings:` / `Database:StartupRetry:` / `Telemetry:` / `Claude:TimeoutSeconds`).

Run (**Windows PowerShell**): `dotnet test tests\WxServices.Setup.Tests\WxServices.Setup.Tests.csproj`. **Expect:** all green.

## 1. Functional isolated-run (manual/scripted; a real SQL Server — no SQLite, `CREATE LOGIN` is instance-level)

Runs the assembled script end-to-end against **throwaway targets only**, then tears them down. Nothing touches the production `WeatherData` / `wxservices` / `C:\HarderWare` / real `services/`.

Set up throwaway targets: a temp services dir with four subfolders + copied `.example` files, and a temp install root:
```powershell
$svc = "$env:TEMP\wx314-services"; $root = "$env:TEMP\wx314-root"
'wxparser','wxreport','wxmonitor','wxvis' | % { New-Item -ItemType Directory -Force "$svc\$_" | Out-Null
  Copy-Item "services\$_\appsettings.local.json.example" "$svc\$_\appsettings.local.json.example" }
New-Item -ItemType Directory -Force $root | Out-Null
```

1. **Run the script** against the isolated set (Windows PowerShell):
   ```powershell
   dotnet run --project src\WxServices.Setup -- --mode full `
     --install-root $root --services-dir $svc `
     --database WeatherDataTest --sql-login wxservicestest --server .\SQLEXPRESS
   ```
   Enter a throwaway password at the prompt.
   - *Expect:* the prerequisite gate reports all-green (bulleted), the printed `CREATE LOGIN`/`CREATE USER`/`GRANT` statements appear and are confirmed, and it completes without a UAC prompt.
2. **Five files written** — *Expect:* `$svc\<svc>\appsettings.local.json` exists for all four (each `ConnectionStrings:WeatherData` names `WeatherDataTest` + `wxservicestest`, `_README`/`Telemetry`/tool-paths preserved), and `$root\appsettings.local.json` holds the native Trusted string.
3. **Login authenticates over TCP** — dial the endpoint the *host* can reach. `host.docker.internal`
   is the **container→host** route only: it resolves inside a container but is not reachable from the
   host itself, so using it here fails with a connection timeout that looks like a defect and is not
   one. (Corrected 2026-07-20 after the first run of this procedure failed on exactly that.)
   ```powershell
   sqlcmd -S "localhost,1433" -U wxservicestest -P "<pw>" -d WeatherDataTest -C -Q "SELECT DB_NAME();"
   ```
   - *Expect:* returns `WeatherDataTest` — proving TCP/1433, Mixed-Mode authentication, the password,
     and the least-privilege roles all work together for that login.

   Then confirm the container-side name in the generated connection string does resolve:
   ```powershell
   docker exec services-wxparser-1 getent hosts host.docker.internal
   ```
   - *Expect:* an address. (The route itself is proven continuously in production — the four service
     containers reach host SQL this way on every cycle.)
4. **Schema + seed** —
   ```powershell
   sqlcmd -S .\SQLEXPRESS -E -C -d WeatherDataTest -Q "SELECT [Key],[Value] FROM dbo.Config ORDER BY [Key];"
   ```
   - *Expect:* the foundational rows present (home/region/bbox/map-extent); the schema exists (Config + the rest).
5. **Re-run idempotency** — run step 1 again.
   - *Expect:* no duplicate-key errors; files overwritten, `Config` rows upserted, login skipped/updated.
5a. **Missing template aborts before any mutation** (regression guard, added 2026-07-20 from the
   `/code-review` pass). Drop the test DB + login, delete one template, and re-run step 1:
   ```powershell
   Remove-Item "$svc\wxvis\appsettings.local.json.example" -Force
   sqlcmd -S .\SQLEXPRESS -E -C -Q "IF DB_ID('WeatherDataTest') IS NOT NULL BEGIN ALTER DATABASE WeatherDataTest SET SINGLE_USER WITH ROLLBACK IMMEDIATE; DROP DATABASE WeatherDataTest; END; IF SUSER_ID('wxservicestest') IS NOT NULL DROP LOGIN wxservicestest;"
   ```
   - *Expect:* exits 1 with `Template not found: ...` naming the path and `--services-dir`, **and**
     `WeatherDataTest` / `wxservicestest` are both still **absent** — the file plan is built before
     anything mutates, so a bad `--services-dir` cannot leave a provisioned login and migrated
     database with no config files.

5b. **The TCP probe follows `--server`** (regression guard, same origin). Re-run step 1 with
   `--server NOSUCHHOST01`.
   - *Expect:* the TCP check names `NOSUCHHOST01:1433` — *not* `127.0.0.1` — and the Mixed-Mode and
     sysadmin checks appear as `not checked — SQL Server was unreachable` rather than silently
     vanishing from the report. Exits 1 having changed nothing.

6. **Tear down (no residue):**
   ```powershell
   sqlcmd -S .\SQLEXPRESS -E -C -Q "DROP DATABASE WeatherDataTest; DROP LOGIN wxservicestest;"
   Remove-Item -Recurse -Force $svc, $root
   ```
   - *Expect:* the real instance shows **no** `WeatherDataTest` / `wxservicestest`, and the real `C:\HarderWare` + `services/` are untouched.

---

**PASS** = §0 unit tests green (CI) **and** §1 isolated-run steps 1–5 meet their Expects with step 6 leaving no residue. Then set Test Result `PASS <date>` → Done.
