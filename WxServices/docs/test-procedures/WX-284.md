# WX-284 — Recipient precipitation terminology collapse: post-deploy functional test

All three steps are deterministic and fully covered offline: the renderer vocabulary and the "Why this
update:" label by the re-recorded goldens, the change-significance behavior (possible↔likely and a
severe tier-bump are non-changes; severe onset/clearing still sends) by the `DeterministicChangeDetector`
and `ChangeConsistencyValidator` tests, and the reframed `changeSummary` rule by the prompt-lock test. So
this procedure verifies only the two things the offline suite **cannot** cover:

1. the **data-only migration** (`WX284RelabelChangeBand`) actually applied to the live DB, and
2. the **deployed build** renders the new vocabulary / label in a real report, and the **live LLM** honors
   the reframed `changeSummary` contract.

Proportionate to a golden-locked rendering change — a smoke that the deployed system behaves, not a live
matrix. Do **not** force an unscheduled send to exercise the band; ride the next natural report cycle.

## 1. Migration applied (immediate — no waiting)

After the release deploys and EF migrations run, confirm the label token relabeled. From PowerShell:

```powershell
sqlcmd -E -C -d WeatherData -Q "SELECT lt.Phrase FROM LanguageTemplates lt JOIN Languages l ON l.Id = lt.LanguageId WHERE l.IsoCode = 'en' AND lt.Token = 'WhatsChangedLabel';"
```

*Expect:* **`Why this update:`** (not `What's changed:`). This is the one hard live fact the goldens don't
touch — they parse the migration source, not the deployed DB.

> Note (en-only seeding, WX-251): only the `en` row is relabeled here. Any live **curated non-en** label
> (e.g. a Spanish equivalent) is stale until the WX-250 top-up regenerates it from the new `en` baseline or
> QA re-curates it. Flag this to the QA pass; it is expected, not a defect.

## 2. Vocabulary smoke — next report with precipitation

On the next scheduled report cycle that carries precipitation, inspect a rendered report body (the delivered
email, or `CommittedSend.EmailBody` in the DB) and confirm the per-day forecast grid uses the collapsed
vocabulary:

- non-severe liquid precip reads **"rain possible"** or **"rain expected"** — never **"likely"**, never
  "showers"/"drizzle"/"downpour"/"thunderstorm";
- any `SevereFlag` block reads **"severe storms possible"** / **"severe weather possible"** — never
  "likely" or "expected";
- frozen precip (snow / wintry mix / freezing rain) still reads its own words.

*Expect:* the above; the word "likely" does not appear on a non-severe precip band, and no severe band
reads "expected".

## 3. Change band — opportunistic (not a blocker)

When the next **unscheduled** send (or a scheduled report fronting a near-term severe onset) occurs during
the observation window, confirm the amber band:

- is labeled **"Why this update:"**;
- leads with the **single triggering change** (the reason for the interruption), not an enumeration of
  every difference.

If no such send occurs in the window, record this step **deferred** — the offline prompt-lock test and the
goldens carry it; it does not gate Done.

## 4. Log smoke

Grep the WxReport service log over the observation window for reconciler degrades:

```powershell
Select-String -Path C:\HarderWare\Logs\wxreport-svc.log -Pattern "degrade|NarrativeProseException|ChangeConsistencyException"
```

*Expect:* no **new** degrade class introduced by the `changeSummary` reframe (the pre-existing WX-177/139
degrade families are unrelated and keyed on their own strings).

## Result

`PASS` requires **all three** of: **step 1** confirms `Why this update:`, **step 2** is clean, and
**step 4** shows no new degrade class. **Step 3** is recorded as *observed* or *deferred* and does **not**
gate the result. If step 4 surfaces a new reconciliation degrade introduced by this change, the result is
**FAIL** — investigate before recording anything. On PASS, set the ticket **Test Result** =
`PASS <date> @ v1.48.0`, then take In Test → Done.
