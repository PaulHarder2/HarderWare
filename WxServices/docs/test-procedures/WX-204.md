# WX-204 / WX-205 / WX-206 — reconciliation degrade + wasted-Claude-call fixes

Shipped together in **1.34.1** under the cost/scalability epic **WX-122**. Verified live by
`WX-204-verify.sh` (one combined script — the three fixes share a failure signature in the log).

## What broke (the analysis, 2026-06-19)

A post-1.34.0 log analysis of reconciliation errors found recurring **degrades** dominated by a
phantom **"Strengthening"**:

> structured_report change 'ch2' (Thunderstorm Strengthening, …) is **not a real change versus
> prior_snapshot**: in-window expectation moved prior→Certain→new→Certain and severeFlag
> prior=True/new=True do not support 'Strengthening'.

Since WX-189, `changes[]` is computed deterministically by `DeterministicChangeDetector` **after**
the Claude call, and `ValidateChangeSnapshotConsistency` checks it as its tautological inverse. They
had **diverged on granularity**: the detector classifies **per block** (and groups consecutive
blocks into one window), while the validator compared the **window aggregate**. A real interior-block
rise — block-0 expectation `Likely→Certain` *and* block-6 severe `False→True`, where block-6 was
already `Certain` and block-0 already severe — leaves the window max/severe **flat**, so the
validator false-rejected it. Each occurrence wasted **3 Claude calls** (the retries can't fix a
change Claude didn't author) and degraded the report.

## The fixes

- **WX-204** — the consistency check now backs a change **per block** (mirrors the detector), so the
  defense-in-depth net is tautological again while still rejecting a true phantom (no block moved).
  Regression test: `DeterministicChangeDetectorTests.Strengthening_FromInteriorBlockRise_…_WX204`.
- **WX-205** — a `ChangeConsistencyException` (a computed-change fault) now **degrades on attempt 1**,
  no 2nd/3rd Claude call, with a loud ERROR pointing at the detector. Cuts per-cycle cost 3 → 1 on
  such faults. *(No end-to-end unit test: post-WX-204 the exception is unreachable from valid detector
  output — it's pure defense-in-depth for a future detector regression — so triggering it through the
  reconcile flow would need a test seam not worth adding; verified by inspection + the WX-204 regression.)*
- **WX-206** — the closing-claim guard's `NarrativeProseException` message now **names the exact
  offending sentence**, so the retry feedback pins it. *(The section-degrade routing it asked for was
  already present — `ValidateClosingClaims` already throws `NarrativeProseException`; the wholesale
  closing degrades were the same WX-204 detector bug re-failing the section-degrade re-validation, so
  WX-204 fixes those too.)*

None of this was introduced by 1.34.0 — it is WX-189/148/151 machinery, untouched by WX-171.

## Post-deploy verification (live)

`WX-204-verify.sh` (pinned to **1.34.1**), since the deploy boundary:

- **Failure signature** — *no* `… is not a real change versus prior_snapshot` line (the phantom
  rejection). Post-fix this should be **0**; any occurrence ⇒ **FAIL** (the per-block fix is
  incomplete or the detector has another disagreement).
- **Gate** — no time window (deterministic fix); PASS needs ≥ 3 reconciliations since deploy (a
  `reconciled` token-log line marks one), so the path is exercised across more than one cycle.
- **Non-gating evidence** (glance per this doc): any **WX-205** immediate-degrade line
  (`degrading immediately without retry`) should *not* be paired with retry attempts; any **WX-206**
  closing-claim fault now carries `sentence: "…"`.

**Run** (bash/WSL, after deploy): `docs/test-procedures/WX-204-verify.sh` (`--since` to re-anchor;
`-h` for the full header). The same standing delegation as the other verify scripts applies — run it
once ≥ 3 reconciliations have accrued, record the Test Result, and move the tickets to Done on PASS.
