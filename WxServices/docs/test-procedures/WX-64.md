# WX-64 Manual Test Procedure — Containerized WxReport.Svc

**What this verifies:** `WxReport.Svc` runs correctly as a Linux Docker container
(`services/docker-compose.yml`, service `wxreport`) instead of a Windows service — it
builds, reaches the host SQL Server, starts, reads its DB-sourced secrets, writes its log
**and heartbeat** through the bind mount where WxManager/WxMonitor read them, can read the
`plots` mount that supplies meteograms, sends Paul its startup **Diagnostic report** (the
end-to-end proof — DB read, Claude call, meteogram inline, SMTP send in one email), and exports
telemetry. The heartbeat + plots checks
are the end-to-end proof of the WX-64 **InstallRoot single-source-of-truth** fix
(`WxConfigurationExtensions.AddInstallRoot`): the workers resolve those paths through
`IConfiguration["InstallRoot"]`, which must equal the container's `/opt/wxservices` root, not
the baked-in `C:\HarderWare`. The build/pattern pieces (image builds, the env seam, the banner)
are build-/unit-covered; this document is the verification record for the live container run.

**Recording results:** when every section passes, set the ticket's custom fields:
**Test Proc** = `docs/test-procedures/WX-64.md` and **Test Result** =
`PASS yyyy-mm-dd @ <version>`. If anything fails, set
`FAIL yyyy-mm-dd @ <version> — section N` and describe it in a ticket comment.

"**Expect:**" lines are pass/fail criteria — if reality differs, note the section and
continue; the list of failed sections is the run's deliverable.

---

## 0. Setup / precondition

1. Confirm Docker Desktop is up and the real (gitignored) config is in place:

   ```powershell
   docker info | Select-String 'Server Version'
   Test-Path C:\Code\HarderWare\services\wxreport\appsettings.local.json
   ```

   **Expect:** a server version prints; the path test returns `True`. (Copy
   `appsettings.local.json.example` → `appsettings.local.json` and fill the SQL password from
   RoboForm if it is not there yet. No API keys go in this file — WxReport reads SMTP/Claude/Gemini
   secrets from the `GlobalSettings` DB row.)

## 1. Build + start the container

```powershell
cd C:\Code\HarderWare\services
docker compose up -d --build wxreport
docker compose ps
```

**Expect:** the image builds and the container starts with no error; `docker compose ps`
shows `wxreport` as `running`/`Up`.

## 2. "Application started" banner (reached the DB, read config, started)

```powershell
docker compose logs wxreport | Select-String 'starting\.','Database ready','Configuration validated','Application started'
```

**Expect:** the log shows `WxReport.Svc 1.52.0 starting.` (version present, and **no**
`(commit …)` fragment), `Database ready.`, `Configuration validated.` (the SMTP + Claude
keys were found in `GlobalSettings` — proving the container read the DB), and
`Application started. Press Ctrl+C to shut down.`. The banner proves `EnsureSchemaAsync`
succeeded against `host.docker.internal,1433`.

## 3. Log + heartbeat through the bind mount (the InstallRoot seam proof)

```powershell
$InstallRoot = (Get-Content 'C:\Code\HarderWare\WxServices\appsettings.shared.json' -Raw | ConvertFrom-Json).InstallRoot   # resolved from shared config, not hardcoded
Get-Content "$InstallRoot\Logs\wxreport-svc.log" -Tail 8
Get-Content "$InstallRoot\Logs\wxreport-heartbeat.txt"
(Get-Item "$InstallRoot\Logs\wxreport-heartbeat.txt").LastWriteTimeUtc
```

**Expect:** `{InstallRoot}\Logs\wxreport-svc.log` accumulates fresh lines from the container —
including a recent `Report cycle complete` / `Next report check in 5 minute(s)`, proving the
worker runs — and `wxreport-heartbeat.txt` holds a **recent** UTC timestamp (within the last
cycle). The heartbeat landing here (not in a `C:\HarderWare`-shaped path inside the container)
is the proof that `IConfiguration["InstallRoot"]` resolved to `/opt/wxservices` — i.e. the
`AddInstallRoot` fix works. **If instead the container logs `heartbeat` under a `C:\HarderWare`
path, section 3 FAILS** (the seam regressed).

## 4. Plots mount readable (meteograms resolve in-container)

```powershell
docker compose exec wxreport ls /opt/wxservices/plots | Select-Object -First 5
# read-only probe: a write MUST be refused
docker compose exec wxreport sh -c 'touch /opt/wxservices/plots/.ro_probe 2>&1; echo exit=$?'
```

**Expect:** the `ls` lists the host's meteogram PNGs, **and** the `touch` fails with a
`Read-only file system` error (non-zero exit) — proving the mount is genuinely `:ro`, not just
readable. So when a report
actually sends, `ReportWorker` (via `WxPaths(_config["InstallRoot"]).PlotsDir`) finds the
meteogram to inline. (A report send is gated by the significance/debounce logic, so an
inlined meteogram may not appear on every cycle; the readable mount + the section-3 heartbeat
together prove the path resolution. If a report does send during the run, confirm the email
carries its meteogram and the log shows no `meteogram … not found`.)

## 5. Diagnostic (startup) report email received — the end-to-end proof

On startup `WxReport.Svc` sends Paul the startup **diagnostic** report
(`SendStartupReportAsync` → `sending startup (diagnostic) report to … (Paul) only`). One email
exercises the whole containerized pipeline at once: the DB read (recipients + the `GlobalSettings`
SMTP/Claude secrets), the Claude generation call (outbound internet from the container), the
meteogram inline (the read-only `plots` mount, section 4), and the SMTP send.

```powershell
docker compose logs wxreport | Select-String 'startup .diagnostic. report','report sent'
```

**Expect:** within a few minutes of container start, **Paul receives the diagnostic report email**,
and the container log shows `sending startup (diagnostic) report to … (Paul) only` followed by
`startup (diagnostic) report sent`. The received email — with its meteogram — is the single
strongest confirmation that the container is a working drop-in for the Windows service.

## 6. Telemetry to the collector / Grafana

Prerequisite: telemetry enabled — `Telemetry:Enabled=true` and the container's
`Telemetry:OtlpEndpoint` ends in `/v1/metrics`. **If telemetry is disabled, skip this section.**
Open Grafana (`http://localhost:3000`).

**Expect:** metrics from the `wxreport` container appear (it reaches the collector at
`host.docker.internal:4318`). A short delay after startup is normal.

## 7. WxMonitor heartbeat watcher repaired (folded-in WX-63 regression)

WX-64 also adds `AddInstallRoot` to `WxMonitor.Svc`, so **the wxmonitor container must be
rebuilt** for this to take effect. In a two-stage deploy (`.\Deploy-WxService.ps1 WxReportSvc`
first, then `.\Deploy-WxService.ps1 all`), **only the `all` stage rebuilds the monitor** — so run
this check after that stage. After stage 1 the monitor still logs the false `heartbeat file not
found` for wxreport (it is still running its old image); verify wxreport's heartbeat on the host
per section 3 instead.

```powershell
cd C:\Code\HarderWare\services
docker compose up -d --build wxmonitor
Start-Sleep 60
docker compose logs --since 2m wxmonitor | Select-String 'heartbeat file not found'
```

**Expect:** after the rebuild, the monitor **stops** logging `heartbeat file not found` for
services whose heartbeat files exist and are fresh (before the fix it logged that every cycle
for every watched service, because it resolved the path under a non-existent in-container
`C:\HarderWare`). A service that is genuinely down may still legitimately show the WARN — the
point is that a *healthy* service no longer does.
