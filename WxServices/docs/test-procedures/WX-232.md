# WX-232 — Translation-QA per-check subfolders + structured file names

Verifies the producer (TranslationQa tool) writes each check into its own `<iso>.<stamp>` subfolder with
`<lang>.<stamp>.<purpose>.<ext>` names, and the consumer (`JudgePackageStore.Discover` → WxManager review
tab) reads the new layout. The discovery/join logic is also covered by automated tests
(`WxReport.Tests/TranslationQaJudgePackageTests.cs`); this procedure covers the producer + the UI.

**Build/run:** `dotnet build WxServices.sln`; WxManager exe at
`C:\HarderWare\BuildCache\WxServices\WxManager\bin\Debug\net8.0-windows\WxManager.exe`.

## Steps

1. **Producer — generate a check** (uses live Claude + Gemini): `dotnet run --project
   src\WxReport.Tools.TranslationQa -- --lang <iso> --judge gemini`.
   *Expect:* a new subfolder `C:\HarderWare\translation-qa\<iso>.<stamp>\` containing:
   - `<iso>.<stamp>.request.md`, `<iso>.<stamp>.request.json`, `<iso>.<stamp>.judged.json`
   - per scenario, `en.<stamp>.<scenario>.html` (reference) and `<iso>.<stamp>.<scenario>.html` (target)
   - (a manual run *without* `--judge gemini` also writes `<iso>.<stamp>.response.txt` — the paste cue-card — into the subfolder)
   - **nothing** written to the flat `translation-qa\` root.

2. **Consumer — WxManager** → **Translation QA** tab → **Refresh**.
   *Expect:* the new `<iso> · <stamp>` package appears in the selector and loads fully (trust bar,
   per-scenario triptych, findings, vocabulary) from its subfolder.

3. **Consumer regression** — the previously migrated packages (de/es/eo) still appear and load with all
   elements in place.
   *Expect:* all packages present; switching between them works.

4. **Manual `--response` path** (no API) — `dotnet run --project src\WxReport.Tools.TranslationQa --
   --response "C:\HarderWare\translation-qa\<iso>.<stamp>\<iso>.<stamp>.response.txt"`.
   *Expect:* the parsed `<iso>.<stamp>.judged.json` is written **into the same subfolder** as the response
   file (not the flat root).
