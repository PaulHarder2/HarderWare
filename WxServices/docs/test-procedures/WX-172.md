# WX-172 — Test Procedure

**Change:** Generation-on-enable. Enabling a supported language no longer requires pre-existing
templates — it marks the language PENDING, and WxReport.Svc on its next cycle calls Claude to
translate the baseline (en) report vocabulary into that language, with a per-token
**representability self-check** and **fail-closed** validation. The language resolves to
**READY** (assignable), **BLOCKED** (a token can't be expressed by substitution — needs a
renderer/code change, not retried), or **FAILED** (a transient error — retried next cycle).
Recipients may only be assigned a READY language. The Languages tab shows each supported
language's status and the operator guidance.

**Ships in:** WxReport.Svc (`TemplateTranslator`, `TranslatorPrompts`, `ClaudeClient.InvokeTranslationAsync` via the shared `SendForToolUseAsync`, the `ReportWorker` cycle hook) + WxManager (Languages/Recipients tabs) + the `BackfillSeededLanguageReady` migration, release **1.35.0**.

**Test Proc field:** `docs/test-procedures/WX-172.md`
**Test Result field:** `PASS yyyy-mm-dd @ 1.35.0` / `FAIL …` (recorded after the §3 in-production run).

---

## 1. Unit tests (pre-merge — the correctness proof)

`TemplateTranslatorTests` and `LanguageStateTests` in `tests/WxReport.Tests`. Run:

```bash
dotnet test WxServices/tests/WxReport.Tests/WxReport.Tests.csproj --filter "FullyQualifiedName~TemplateTranslatorTests|FullyQualifiedName~LanguageStateTests"
```

**Expect:** all pass. The suite covers, directly:

- **Validator accepts** a complete well-formed set, a not-representable token carrying a note
  (BLOCKED), and placeholders reordered for grammar.
- **Validator rejects** (fail-closed, JsonException → retry/Failure, never persisted): a missing
  token, an unknown extra token, a duplicate token, a dropped `{n}` placeholder, an empty phrase
  on a representable token, a not-representable token with no note, a control character in a
  phrase **or in a blocked token's context** (both paths), an over-length phrase, and an
  empty/over-length culture tag.
- **State model** — the `(GeneratedAtUtc, GenerationError)` encoding maps to
  Disabled/Pending/Ready/Blocked/Failed, and `IsReady` is true only for READY.

## 2. Migration check (on the first start AFTER deploy)

The `BackfillSeededLanguageReady` migration ships **inside** the 1.35.0 build and is applied by
`DatabaseSetup.EnsureSchemaAsync` at service startup — so it runs on the **first start after the
1.35.0 deploy**, never before. It stamps `GeneratedAtUtc` on every enabled language whose
representable templates already cover the full baseline (en + es today), so the new generation
cycle does not re-translate the seed.

Run this check **after** the deploy + first start (before deploy, en/es correctly still show
`GeneratedAtUtc` NULL — PENDING):

```bash
sqlcmd -S .\SQLEXPRESS -d WeatherData -E -C -Q "SELECT IsoCode, IsEnabled, GeneratedAtUtc, GenerationError FROM Languages WHERE IsEnabled = 1"
```

**Expect:** `en` and `es` both `IsEnabled=1`, `GeneratedAtUtc` **non-null**, `GenerationError`
NULL (READY). No regeneration of en/es appears in the WxReport log after deploy.

## 3. Functional verification (post-deploy — operator-triggered, MANUAL)

This feature's behaviour is triggered by an operator action (enabling a language), so the
end-to-end confirmation is performed **by hand** once after deploy (steps below). The automated
`WX-172-verify.sh` (§4) then asserts the resulting data invariants — but it is the **final** step,
run only after these manual steps pass.

1. **Enable a new language.** In WxManager → Languages, select a language with no templates yet
   (e.g. *French*) and click **Enable**.
   **Expect:** status message "…queued for generation on an upcoming report cycle…"; the language moves to
   Supported showing **[Generating…]**; selecting it shows the "Generating —" guidance.
2. **Wait one report cycle** (≤ the `Report:IntervalMinutes`, default 5 min). In the WxReport log:

   ```text
   WX-172: generating templates for 'fr' (French)…
   WX-172: 'fr' generated and READY — N token(s).
   ```

   **Expect:** the language now shows **[Ready]** in the Languages tab (reload the tab); the
   detail panel reads "Ready — generated … UTC."; `Languages.GeneratedAtUtc` is set,
   `GenerationError` NULL; `LanguageTemplates` has a full row set for it.
3. **Recipient gating.** On the Recipients tab, the newly-READY language now appears in the
   language dropdown; a still-generating (PENDING) language does **not**.
   **Expect:** only READY languages are assignable.
4. **Cost visibility.** The generation's token usage appears on the Claude cost counters (the
   `wxreport.claude.tokens.*` meters) and a `translate:fr` DEBUG line in the log.
   **Expect:** the spend is recorded, not invisible.
5. **(If observed) BLOCKED.** Should a language come back BLOCKED, the log reads
   `WX-172: 'xx' BLOCKED — k of N token(s) not representable: …`; the tab shows **[Blocked]** with
   the offending token(s) + note and the "click Disable to shelve it" guidance; the language is
   **not** assignable and is **not** retried on subsequent cycles.

> Generation runs **once per cycle at most** (after the sends), so enabling several languages at
> once drains one per cycle — expected, since a not-yet-READY language has no recipients.

## 4. Automated invariant check — `WX-172-verify.sh` (FINAL step)

> **Do not run this step unless every check in §3 (steps 1–5, as applicable) passed.** It is the
> closing confirmation, not a substitute for the manual walkthrough: it asserts the *data* is
> self-consistent, but only the manual steps confirm the *rendered/observed* behaviour. If any §3
> step failed, stop and record `FAIL` — do not run the script to "see if it passes anyway."

Once §3 is clean, run the automated check against the live DB + log:

```bash
bash WxServices/docs/test-procedures/WX-172-verify.sh
```

It resolves the deploy boundary version-pinned (`deploy-info.sh --version 1.35.0 WxReportSvc`) and
asserts, from `WeatherData`, that **every** WX-172 state invariant holds (each count must be **0**):

- **`ready_incomplete`** — a READY language **missing ≥1 representable baseline (en) token**. This
  is the "all needed templates are present" check: a language marked READY must carry the full set.
- **`ready_with_blocked`** — a READY language that nonetheless has a `Representable=0` row (it
  should have been BLOCKED).
- **`blocked_without_rows`** — a BLOCKED language whose block reason has no `Representable=0` row.
- **`recipients_not_ready`** — a recipient assigned a **non-READY** language (the Recipients-tab
  gate must hold in the data too; a NULL assignment is the service default and is fine).
- **`ready_no_culture`** — a READY language with no `CultureName`.

It also reports the state census (enabled/ready/blocked/failed/pending) and, since the deploy
boundary, the report-cycle count (PASS gate: ≥ 2), the generation log fingerprints
(READY/BLOCKED/FAILED/threw), and the background-ERROR delta vs the equal-length pre-deploy window.

**Expect (PASS):**

```text
                                          total : 0   (expect 0 -- the failure signature)
   ====>  PASS   WX-172 verified live: …
```

- `WAIT` is returned in two cases: **(a)** before the version is deployed, or before any service-log
  line exists since the boundary — here the script exits *before* the DB query runs; **(b)** once the
  service has logged but fewer than 2 report cycles have, where the DB invariants **were** evaluated
  and are currently clean but the post-deploy cycle evidence is still insufficient. A nonzero invariant
  count **always FAILs immediately**, regardless of the cycle count. Re-run once cycles have logged.
- Any nonzero invariant total: `FAIL` — inspect the named count (a missing-token READY language
  means generation stamped READY without a full set; a non-READY recipient means the gate or a
  re-enable left a recipient stranded).

Record the verdict's `Test Result` string (`PASS yyyy-mm-dd @ 1.35.0` / `FAIL …`) in the Jira
**Test Result** field.
