# WX-139 Functional Test Procedure — synoptic-mechanism grounding (post-deployment verification)

**What this verifies:** that WX-139 (≥ 1.47.1) is live and that **live Claude complies**
with the mechanism-grounding rule — reader-facing prose states *what* changes but never
attributes a synoptic *cause* ("…as a front pushes through") it cannot evidence from
single-point data. Two layers ship: a **prompt rule** (`ReconcilerPrompts`) that tells
Claude to describe the effect and never the cause, and a **deterministic validator**
(`CheckProse` in `ForecastReconciler`) that fails closed through the WX-189 retry if a
mechanism term slips through. The unit tests prove the validator; only production proves
that live Claude, under the new prompt, actually stops inventing mechanisms — so the
success state here is *silence* (clean reports, no exhausted mechanism degrades).

**Recording results:** on pass, set the ticket's custom fields — **Test Proc** =
`docs/test-procedures/WX-139.md`, **Test Result** = `PASS yyyy-mm-dd @ <version>`. On
failure: `FAIL yyyy-mm-dd @ <version> — <which check>`, a ticket comment describing it,
and open the Bug (the `FAIL: Open bug ticket` transition).

---

## ⚠️ Do not confuse WX-139 with the pre-existing precip-at-a-dry-time degrades

A separate, **already-live** reconciliation issue produces `ValidateClosingClaims`
degrades — the closing prose asserts *precipitation/storm at a time the snapshot is dry*
(WX-177 territory). It shares WX-139's **exact WX-189 wrapper**:

```
ForecastReconciler.cs::ReconcileAsync:480 — Reconciliation could not make the Closing
prose self-consistent after 3 attempts (…)
```

So a naive "watch the reconciler log for degrades" check would false-count those as
WX-139 failures. The distinguishing signal is the **inner message**:

| Defect | Inner string to grep | Ticket |
|---|---|---|
| WX-139 (this) | `attributes a synoptic mechanism` | WX-139 |
| precip-at-dry-time | `asserts precipitation/storm activity at a local time` | WX-177 |

The verify script keys on the WX-139 inner string only, and reports the WX-177 count
separately (labelled "NOT WX-139"), so the two are never conflated.

## 0. Run the verification

The check is a **bash** script (not PowerShell) — `WX-139-verify.sh`, beside this
procedure — that reads the WxReport service log and `deploy-history.log`. It needs bash
with read access to those logs; on the HarderWare PC that means **WSL**. Run from the
repo root:

```bash
bash WxServices/docs/test-procedures/WX-139-verify.sh
```

From **PowerShell**, shell out to WSL (also from the repo root):

```powershell
wsl bash WxServices/docs/test-procedures/WX-139-verify.sh
```

No arguments needed: the script is **version-pinned** — it embeds the release `VERSION`
it tests (`1.47.1`) and its `COMPONENTS` (`WxReportSvc`), and asks the shared
`deploy-info.sh` helper for the most recent `deploy-history.log` entry matching that
version, getting back the boundary timestamp and the deployed commit. If `1.47.1` is not
deployed yet, the verdict is **WAIT** (nothing to verify). Pass `--since` to override the
boundary manually only if you know what you are doing (an earlier boundary pulls the old
binary's output into the window).

## 1. Re-record the KDWH replay fixtures (live Claude compliance, one-time)

The prompt changed materially, so the KDWH replay fixtures must be re-recorded so they
remain genuine captures of Claude's judgment under the *current* prompt (fixtures
README). This is a **live, billable Anthropic call** and needs the API key. Run it once
per the **Re-recording** section of the fixtures README
(`tests/WxReport.Tests/Scenarios/Fixtures/README.md`) — it uses a `.runsettings` file,
because the inline / `export` env forms silently no-op under .NET SDK 9 (the `dotnet
test` host doesn't inherit the shell environment; WX-285).

**Expect:** the recorder overwrites `kdwh-853-*.recorded.json` + `kdwh-853-skip.trace.txt`,
and the replayed `KdwhScenarioReplayTests` still pass against the fresh captures. Commit
the overwritten fixtures. If real Claude's skip/send judgment changed, investigate before
trusting it (README).

## 2. Confirm the rule is live and Claude complies

Run the script (step 0) after at least one **full active day** of report cycles has
accrued past the `1.47.1` deploy boundary.

- **Expect:** *Reports flowed since deploy* > 0 (the fix is live and producing reports).
- **Expect:** *EXHAUSTED synoptic-mechanism degrades* = **0**. This is the PASS signal —
  live Claude, under the new prompt, does not attribute a cause it cannot evidence.
- *Self-correcting mechanism retries* ≥ 0 is **fine** — a nudge that Claude then obeys is
  the prompt+retry loop working, not a failure.

## 3. If an exhausted mechanism degrade fired — adjudicate it

The script prints the flagged lines. Read each:

- **Genuine mechanism** (the prose really did name a front / pressure system / etc.):
  live Claude is not fully complying → **FAIL**; open a Bug to strengthen the prompt rule
  (this is the prompt layer under-performing; the backstop still protected the recipient).
- **False positive** (the flagged prose is legitimate — e.g. a positional "in front of",
  or an innocent use the regex over-matched): → **FAIL**; open a Bug to tighten the regex
  (`SynopticEnglishMechanism` / `SynopticSpanishMechanism`).

## 4. Spot-check a shipped report (belt-and-suspenders)

Read one or two recently delivered reports (any enabled language). **Expect:** the change
band and closing describe observed effects ("winds turn gusty with showers arriving this
evening") with **no** invented cause. Because the validator fails closed, a mechanism can
never reach a recipient — this step confirms the prose still reads naturally without the
banned "why," not that a mechanism leaked.

---

**PASS** = KDWH fixtures re-recorded and replays green (step 1); reports flowed with zero
exhausted synoptic-mechanism degrades over a full active day (step 2); no unadjudicated
flagged lines (step 3); shipped reports read clean (step 4). The verify script's WAIT /
PASS / FAIL verdict encodes steps 2–3 and prints a ready-to-paste Test Result string.
