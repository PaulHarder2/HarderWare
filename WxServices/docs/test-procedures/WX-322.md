# WX-322 Test Procedure — What3Words key moves to `GlobalSettings`

**What this verifies:** that the What3Words API key is read from the `GlobalSettings` database row
rather than `appsettings.local.json`, that it is editable from the Configure tab, and that
`///word.word.word` lookups keep working with **no key in any file**.

**Run it interactively** — read one step, perform it, report the result; note failures and continue.
The list of failed steps at the end is the run's deliverable.

**Recording results:** **Test Proc** = `docs/test-procedures/WX-322.md`; **Test Result** =
`PASS yyyy-mm-dd @ <version>`.

---

## ⚠️ Sequencing — there is a window where lookups are broken

Deploying this build makes `RecipientsTab` read the key from `GlobalSettings`, which is **empty**
immediately after the migration adds the column. The key still sitting in `appsettings.local.json`
is no longer consulted.

So between **step 1** (deploy) and **step 3** (save the key) the `///` form does not work. That is
expected and brief. **Have the key to hand before starting** — it is in
`C:\HarderWare\appsettings.local.json` under `What3Words:ApiKey` until step 5 removes it.

## 0. Unit tests (CI)

`dotnet test WxServices.CI.slnf` — **Expect:** green, including
`MetarParser.Tests.GlobalSettingsSecretsTests` (round-trip, null-allowed, and the guard that every
secret is a column on the row).

## 1. Deploy and confirm the migration applied

Deploy WxManager (elevated Windows PowerShell): `.\Deploy-WxService.ps1 WxManager`.

The EF migration runs at startup. Confirm the column exists:

```powershell
sqlcmd -S .\SQLEXPRESS -E -C -d WeatherData -h -1 -W -Q "SET NOCOUNT ON; SELECT name FROM sys.columns WHERE object_id = OBJECT_ID('dbo.GlobalSettings') AND name = 'What3WordsApiKey';"
```

- *Expect:* `What3WordsApiKey` returned.

## 2. The Configure tab offers the field

Open WxManager → **Configure**.

- *Expect:* a group **Address Lookup (What3Words)** with a masked **What3Words API Key** box, empty
  (the column is new and null).
- *Expect:* nothing else on the tab changed.

## 3. Put the key into the database

Either route works — the column is what matters, not how it is filled.

**Via the Configure tab:** paste the key from `appsettings.local.json` into the field and press
**Save Configuration**.

**Or via SSMS / sqlcmd** (avoids typing a secret into a masked box you cannot read back — see
WX-323):

```sql
UPDATE dbo.GlobalSettings SET What3WordsApiKey = '<key>' WHERE Id = 1;
```

If the row does not exist yet (fresh install), insert it instead:

```sql
IF NOT EXISTS (SELECT 1 FROM dbo.GlobalSettings WHERE Id = 1)
    INSERT INTO dbo.GlobalSettings (Id, What3WordsApiKey) VALUES (1, '<key>');
```

- *Expect:* `"Configuration saved."`
- Confirm it landed (the value is masked in the UI, so check length rather than content):

```powershell
sqlcmd -S .\SQLEXPRESS -E -C -d WeatherData -h -1 -W -Q "SET NOCOUNT ON; SELECT 'len='+CONVERT(varchar(10),LEN(What3WordsApiKey)) FROM dbo.GlobalSettings WHERE Id=1;"
```

- *Expect:* a non-zero length matching your key (8 for a standard What3Words key).

## 4. `///` resolves — from the database, with the file still present

On the **Recipients** tab, enter a `///word.word.word` address you can verify.

- *Expect:* it resolves to coordinates and a locality, and `wxmanager.log` shows
  `resolved '///…' via What3Words → (lat, lon) locality='…'`.
- *If it fails,* the log now names where the key belongs — "the key lives in the GlobalSettings
  database row (column What3WordsApiKey); set it in WxManager → Configure → Address Lookup". The
  pre-WX-322 message cited `What3Words:ApiKey` in `appsettings.local.json`, which is now the wrong
  place to look.
- *Expect:* **no restart was needed** — the key is read at point of use, not captured at startup.

## 5. Remove the key from the file — the decisive step

```powershell
$local = 'C:\HarderWare\appsettings.local.json'
Copy-Item $local "$local.pre-wx322.bak"
$json = Get-Content $local -Raw | ConvertFrom-Json
$json.PSObject.Properties.Remove('What3Words')
$json | ConvertTo-Json -Depth 10 | Set-Content $local -Encoding UTF8
Get-Content $local
```

- *Expect:* no `What3Words` section remains; everything else (`ConnectionStrings`, `Fetch:*`,
  `Claude:Model`, `WxVis:MapExtent`) is untouched.

## 6. `///` still resolves with no key in any file

Repeat step 4.

- *Expect:* it resolves. This is the point of the ticket — the key now lives only in the database.
- Then **restart WxManager** and repeat once more. *Expect:* still resolves.

## 7. No secret remains in a file

```powershell
Select-String -Path 'C:\HarderWare\appsettings.local.json','C:\HarderWare\WxManager\appsettings.shared.json' -Pattern 'ApiKey|Password|What3Words' -SimpleMatch
```

- *Expect:* **no matches in those two files.**

Note what this step does and does not claim. It checks the **WxManager** configuration only. The
four container files (`services/<svc>/appsettings.local.json`) still contain the SQL login password
inside their connection strings — that is the documented exception, since a connection string
cannot live in the store it unlocks. So the correct reading is *"no secret beyond the bootstrap
credential remains in WxManager's configuration"*, not *"no secret exists in any file"*.

---

**PASS** = steps 0-7 meet their Expects. Record `PASS <date> @ <version>` → Done.

**Rollback if anything fails:** restore `appsettings.local.json.pre-wx322.bak`; the old build reads
the key from the file. The database column is additive and harmless if unused.
