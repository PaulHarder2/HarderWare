# WX-239 — Verification: German `through [day]` renders as an inclusive span

**Functional (post-deploy) verification, In Test.** This procedure replaces the earlier *absence-based*
version (which passed on "no watcher findings" and would have **false-PASSed** — silence from a watcher is
not evidence; WORKFLOW-CORE principle 7). WX-239 is a fix-and-verify, not a monitoring setup: the deployed
change must be shown to have *reduced the real defect rate against a measured baseline*, and the per-language
translation QA must show the new tokens are rendered idiomatically.

The change is prompt-glossary anchoring (`span_through` / `span_until` SOFT Hint tokens + `PromptGlossaryToken`
rows). There is **no deterministic gate** for this class (span inclusivity is not recoverable from the target
text alone — decided with Paul), so verification is two complementary evidence checks, not a pass/fail script.

## Baseline (measured pre-fix, 2026-07-21 — do not recompute; this is the number to beat)

Pairing each German report to the English report from the same `ForecastSnapshotId`, over 41 German reports
2026-06-27 → 2026-07-21:

| Verdict | Count |
|---|---|
| **BUG** — English `through X` rendered German `bis X` (truncates span by a day) | **43** |
| ok — `into X` → `bis in den X` | 13 |
| ok — `by X` → `bis X` | 3 |

The method is the sibling-pairing analysis recorded in WX-239's Jira grounding comment of 2026-07-21. The
post-deploy re-measure (step 6) re-runs **that same method** over the post-deploy window and compares.

## Steps

1. **Confirm the release is deployed.** `deploy-history.log` shows `WxReport` at **v1.60.0**.
   *Expect:* a `WxReport` / `1.60.0` / commit / OK line. Record its timestamp as the verification boundary
   (`deploy-info.sh --version 1.60.0`); until it appears the verdict is **WAIT**, not PASS.

2. **Confirm the en tokens seeded and are anchored.** Against `WeatherData`:
   - `SELECT Token, Phrase, ContextKind FROM LanguageTemplates WHERE LanguageId = 37 AND Token IN ('span_through','span_until');`
     *Expect:* two rows, `Phrase` = `through` / `until`, `ContextKind` = `Hint`.
   - `SELECT Token FROM PromptGlossaryTokens WHERE Token IN ('span_through','span_until');`
     *Expect:* both present.

3. **Confirm top-up generated the target-language phrases.** After the services have run several report
   cycles post-deploy (top-up generates one language per cycle — WX-250):
   `SELECT l.IsoCode, t.Token, t.Phrase, t.ReviewedBy FROM LanguageTemplates t JOIN Languages l ON l.Id = t.LanguageId WHERE t.Token IN ('span_through','span_until') AND l.IsoCode IN ('de','es','eo') ORDER BY l.IsoCode, t.Token;`
   *Expect:* `de` (and eventually `es`, `eo`) each acquire a phrase for both tokens — for **de**, `span_through`
   should read `bis einschließlich` (or an idiomatic inclusive form), `span_until` a plain `bis`. A language
   without its phrase yet is **not a failure** (SOFT degrades to free rendering) — it is a WAIT for that
   language's top-up slot, not a FAIL.

4. **Run translation QA per language** (the WX-235 operator-triggered path) for `de`, `es`, `eo`.
   *Expect:* the run completes and produces judge output (back-translation + critique) covering the span tokens.

5. **Paul reviews the judge output for the span tokens** (the primary review instrument — this is a native
   read, not a regex). *Expect:* Paul confirms the de `through`-span renders inclusively (last day included) and
   the `until`-span is not over-precisified. **Regression guard for es/eo:** anchoring `span_through` for *every*
   enabled language risks nudging a language that idiomatically avoids a span preposition toward a literal one —
   Spanish expresses inclusive spans as `del viernes al domingo` (enumeration), **not** `hasta <día>` (the
   grounding analysis found ZERO bare `hasta <día>`). Confirm es/eo did **not** regress toward a literal
   preposition; a language with no natural single-preposition span form should carry a `Note` on its generated
   phrase rather than a forced literal. Record his verdict per language.

6. **Production re-measure vs the 43-occurrence baseline.** After a full active window post-deploy (≥ ~1 week,
   for a comparable German-report count), re-run the **sibling-pairing analysis** from the 2026-07-21 grounding
   comment over reports generated *after* the step-1 boundary, and count English-`through X` → German-`bis X`
   (bare, span-truncating) occurrences.
   *Expect:* the truncating count is **substantially reduced toward zero** relative to the 43-occurrence,
   near-daily baseline. Some residual is expected and accepted (best-effort anchoring, no deterministic gate);
   a return to the baseline rate is a FAIL. Record the post-deploy count and the window it covers.

## Pass / fail

- **PASS → Done:** en tokens seeded and anchored (2), de acquired an inclusive `span_through` phrase (3), the
  QA run + Paul's review are satisfactory (4–5), and the production re-measure shows a substantial drop from
  the 43-occurrence baseline (6). Record `Test Result` = `PASS yyyy-mm-dd @ v1.60.0` with the post-deploy count.
- **FAIL → open a Bug:** the re-measure shows no material reduction, or QA/Paul's review shows the de span is
  still truncating (or the `until` fix introduced an over-exclusive rendering). We fix forward — the residual
  gets its own linked Bug; WX-239 records the FAIL and the count.
