# WX-239 — Translation-error monitoring via the report-error watcher

**Not a fix-and-verify test.** WX-239 (intermittent literal/timing mistranslations — e.g. "through
Friday" → German "bis Samstag") was *deferred*, not fixed: the class is non-deterministic and
LLM-triggered, so QA sampling could not reliably confirm or refute it (0/4 flagged errors reproduced on
2026-07-07). Instead of chasing it by sampling, the **WX-273 report-error watcher** monitors *shipped*
reports for the class's known signatures, and the evidence it accumulates decides WX-239's fate. This
document records that monitoring setup and its pass/fail.

## What is being watched

Operator patterns in `{InstallRoot}\report-error-patterns.json` (re-read each cycle → live reload; edited
on the server, no redeploy, no UI). Configured 2026-07-08 @ v1.46.0:

| id | lang | regex | catches |
|---|---|---|---|
| `wx239-de-praegen` | `de` | `prägen` | "prägen" (characterize/shape) misused for *persist / continue* |
| `wx239-de-bis-samstag` | `de` | `bis Samstag` | "through Friday" → "bis Samstag" (span truncated by a day) |
| `wx239-de-bis-donnerstagmittag` | `de` | `bis Donnerstagmittag` | "through Thursday afternoon" → wrong timing |
| `wx239-da-lufttryk` | `da` | `lufttryk` | "flow" (air movement) → "lufttryk" (air pressure) |

These are **semantic** errors (the right word in the wrong context), so a match is a **review candidate**
— a report that *contains* the phrase — not an auto-confirmed error. Note `lufttryk` is higher-signal than
it appears: we do not forecast barometric pressure, so any occurrence of it in a Danish report is suspect.

## Monitored surface

- **`de` arm** — live against recipient `paul_de` (the one German recipient in the roster).
- **`da` arm** — recipient `paul_da` added 2026-07-08 to activate `lufttryk`. **Temporary**: to be removed
  if no problem surfaces within a week (see below). Danish is a Ready language, but without a `da`
  recipient there are no Danish reports, so the `da` arm is dormant until one exists.
- **Forward-only** from watermark ~`2026-07-08T16:42Z` (`MonitorState.LastReportScanUtc`). The detector was
  calibrated on 2026-07-08 (see `WX-273.md`), so a lack of findings is *trustworthy silence*, not a dead sensor.
- Findings append to `{InstallRoot}\Logs\findings.jsonl` with `patternId` = `wx239-*`.

## Pass / fail (absence-based)

- **Observation window:** ≥ 1 week (opened 2026-07-08; review on/after 2026-07-15).
- **PASS → close WX-239:** no `wx239-*` findings across the window. The class is not recurring in
  production (plausibly already handled by later fixes such as WX-246 day-name anchoring). Remove the
  temporary `paul_da` recipient.
- **Findings → WX-239 needs a real fix:** any `wx239-*` finding is a candidate recurrence. Review each
  flagged report against its English source; if it is a genuine error, WX-239 gets an actual fix (prompt /
  renderer / validator) and is then re-verified. We deferred the fix precisely because we could not
  reproduce it — a finding *is* the reproduction we lacked.

## How to check (on/after 2026-07-15)

1. `Select-String 'wx239-' {InstallRoot}\Logs\findings.jsonl` — ignore any `calibration-*` lines left from
   the 2026-07-08 detector calibration.
2. **Clean** → PASS: close WX-239, remove `paul_da`.
3. **Findings** → for each, pull the `reportId`'s `CommittedSend.EmailBody`, confirm the phrase is actually
   the WX-239 error (not a legitimate use of the word), and open the fix if confirmed.

**Test Result:** `PASS yyyy-mm-dd @ v1.46.0` once the window closes clean; otherwise record the findings and
pivot to fix-and-verify.
