# AI Engineering Landscape Scan — June 2026

## TL;DR

Claude Opus 4.8 (GA May 28) improves long-horizon agentic coding and tool triggering at the same $5/MTok input price as Opus 4.7 — switch Claude Code from 4.7 to 4.8. The WxServices runtime stays on Sonnet 4.6; none of the new Opus 4.8 features compensate for its 67% cost premium over Sonnet 4.6 for single-turn structured tool-use calls.

---

## Actionable Findings

### 1. Claude Opus 4.8 — two verdicts, one release

**What changed.** `claude-opus-4-8` went GA on May 28, 2026. Improvements over Opus 4.7: better tool triggering (fewer skipped required tool calls), better long-horizon agentic coding (fewer derailments after context compaction in long traces), improved reasoning-effort calibration at each effort level. The `effort` parameter now defaults to `high` on all surfaces including Claude Code and the API. Minimum cacheable prompt length drops to 1,024 tokens (lower than on 4.7). Pricing unchanged: $5/MTok input, $25/MTok output.

**Verdict A — Claude Code (coding assistant, currently Opus 4.7).** Upgrade recommended. Same price; materially better tool triggering and compaction recovery directly benefit complex multi-file coding sessions. In Claude Code settings, switch model from `claude-opus-4-7` to `claude-opus-4-8`. If you set `effort` explicitly anywhere it keeps working; if you relied on the default being below `high`, calibrate. The migration guide has a checklist; `/claude-api migrate this project to claude-opus-4-8` automates any API code changes.

**Verdict B — WxServices runtime (`claude-sonnet-4-6`, confirmed via `appsettings.shared.json`).** No change. The runtime is a single-turn structured tool-use call per reconciliation cycle — it doesn't exercise long-horizon agent loops or compaction. Opus 4.8 at $5/MTok vs. Sonnet 4.6 at $3/MTok is a 67% input-cost premium with no commensurate benefit for this workload.

**Cost to add.** Free (existing Claude Code Max plan) for the coding-assistant switch.

**Recommended next step.** Update Claude Code model to `claude-opus-4-8`. No change to `appsettings.shared.json`.

### 3. Claude Code Dynamic Workflows (v2.1.154, May 28) — new orchestration ceiling

**What changed.** In Claude Code, describe a multi-part task and Claude designs and runs a workflow automatically, orchestrating tens to hundreds of subagents. `/workflows` shows run status.

**Why it matters.** The May 2026 scan introduced Agent View + `/goal` for explicit parallelization. Dynamic workflows let Claude design the agent topology itself. For solo work on WxServices — e.g., "write integration tests for all four services" — this could be handed off as a single prompt rather than requiring manual subagent orchestration. This is a new working pattern, not a subscription change.

**Cost to add.** Free — existing plan.

**Recommended next step.** Try on a well-scoped, low-risk task; judge output quality before using on production-adjacent work.

### 4. Cache Diagnostics API (beta) — debugging tool for WxServices ClaudeClient

**What changed.** New Anthropic API beta feature (header: `cache-diagnosis-2026-04-07`). Pass `diagnostics.previous_message_id` on a Messages request; the API returns a `cache_miss_reason` indicating exactly where the prompt prefix diverged from the referenced prior call.

**Why it matters.** `ClaudeClient.cs` already records `cache_read_input_tokens` and `cache_creation_input_tokens`. If cache hit rates ever look unexpectedly low (visible in OTel metrics), this feature gives a direct root cause without binary search. To use the current single-turn pattern, pass the prior response's `id` as `previous_message_id` on the next reconciliation cycle call.

**Cost to add.** Free — add the beta header behind a debug flag in `ClaudeClient.cs` when needed.

**Recommended next step.** No immediate action. Keep as a known tool; add the header only when investigating a cache regression.
